Capstone · End-to-end

A chat platform: HTTP → attention

This is the whole stack in one page. We design a Claude.ai-style threads product — users create threads, type a message, watch a reply stream back token by token — and then we follow a single request all the way down: from the POST that leaves the browser, through the edge, the gateway, the services and their data stores, into a GPU model server, into the attention and MLP math itself, and back up as a stream of Server-Sent-Events. Part A draws the architecture; Part B traces one request through every box. If you can hold both halves in your head at once, you understand how a model becomes a product.

ELI5

Think of the platform as a high-end restaurant. The front door and security are the edge (CDN, TLS, WAF) — they keep the street out and point you to the nearest location. The maître d’ is the API gateway: checks your reservation (auth), makes sure no single table orders the entire menu (rate limits), and hands you a ticket so a repeated order isn’t cooked twice (idempotency). The waiters are the application services: they remember your table’s history (the thread), write down the new order, and carry plates back and forth. The kitchen is the inference layer — an expediter (scheduler) batches tickets and assigns them to stations (GPU servers). And the chef actually cooking — chopping, searing, plating — is the attention and MLP math inside the model. Everything on this page is either the dining room or the kitchen; only the very last step is the chef.

Part A · The architecture

A chat product looks deceptively simple from the outside — a text box and a stream of words. Underneath, every request crosses seven layers, each owning one job and handing off cleanly to the next. Here is the whole spine before we zoom into any one box.

Seven layers. The cheap rejections (bad token, over quota) happen high and early; the expensive work (prefill + decode on a GPU) happens deep and late; tokens stream back up through pub/sub to the client. Cross-cutting concerns — tracing, secrets, feature flags, autoscaling, multi-region — wrap every box.

1 · Clients (web & mobile)

The client renders the threads list (a paginated sidebar of past conversations) and the active thread. When you send a message it does three things at once: optimistically appends your message to the UI immediately (so the app feels instant), opens a streaming connection for the reply (SSE or WebSocket), and shows a typing indicator until the first token lands. It must handle reconnection — mobile networks drop — by remembering the last event id and resuming the stream, and it must reconcile its optimistic state once the server confirms the real message ids.

SSE vs WebSocket: a chat reply is a one-way server→client token stream, so SSE (a long-lived HTTP response of text/event-stream) is the simpler fit — it rides normal HTTP, reconnects natively via Last-Event-ID, and needs no custom framing. WebSocket earns its keep only when you need rich bidirectional traffic (live collaboration, voice).

2 · Edge

Before a request reaches your code it crosses the edge: a CDN serves static assets and the app shell from a POP near the user; TLS termination ends the encrypted connection at the edge; a WAF and DDoS scrubbing drop obvious attacks (SQL-injection probes, volumetric floods) before they cost you a backend thread; geo-routing sends the user to the nearest healthy region; and a load balancer spreads requests across gateway instances with health checks. The edge is your first, cheapest line of defense — the more junk it absorbs, the less your origin pays for.

3 · API gateway / BFF

The gateway is the single choke point every API request passes through, which is exactly why it owns the cross-cutting work. In order, per request:

  • Authentication. Resolve the caller. A session token → a Redis lookup (GET session:<token>) returns the user and tenant; or a signed JWT is verified locally (no round-trip) and trusted until expiry. No identity → 401, nothing downstream runs.
  • Authorization / ACL. Is this user allowed to touch this thread? The deep check happens in the conversation service, but coarse scopes (is the token allowed to write messages at all?) are checked here.
  • Request validation. Reject malformed bodies, oversized payloads, and bad content types before they reach business logic — schema-validate {content, stream}, cap message length.
  • Idempotency. A client retry must not create two messages. The client sends an Idempotency-Key; the gateway records it (SETNX idem:<key>) and replays the stored result on a duplicate.
  • Rate limiting. A token bucket per user/tenant, limiting both requests-per-minute AND tokens-per-minute — because LLM load scales with tokens, not request count. Over limit → 429 + Retry-After.
  • Routing, request-ids & tracing. Stamp a request id, start a distributed trace span, and route to the right service.

4 · Application services (stateless, horizontally scaled)

Behind the gateway sit small, stateless services you can scale independently. Keeping them stateless — all shared state pushed into the data stores — is what lets you run many copies behind the load balancer and lose any one without losing data.

5 · Data stores — and why each one

The single most common design mistake is forcing everything into one database. Each store below is chosen for a property the others lack: durability, speed, blob capacity, or vector search.

PostgreSQL — the system of record. Durable, transactional, relational. Users, threads, messages, and usage live here because you must never lose a message and you need foreign keys, joins, and consistency.

CREATE TABLE users (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  email        text UNIQUE NOT NULL,
  tenant_id    uuid NOT NULL,
  created_at   timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE threads (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid NOT NULL REFERENCES users(id),
  title        text,
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE messages (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  thread_id    uuid NOT NULL REFERENCES threads(id) ON DELETE CASCADE,
  role         text NOT NULL CHECK (role IN ('user','assistant','system')),
  content      text NOT NULL,
  input_tokens  int,
  output_tokens int,
  created_at   timestamptz NOT NULL DEFAULT now()
);

-- the hot query: load a thread in chronological order
CREATE INDEX idx_messages_thread_time ON messages (thread_id, created_at);
CREATE INDEX idx_threads_user_updated ON threads (user_id, updated_at DESC);

Redis — the fast shared brain. In-memory, sub-millisecond, perfect for ephemeral hot state that many stateless instances share. It plays six roles here, each a different key:

session:<token>          → {user_id, tenant_id, exp}     # auth lookup
ratelimit:<user>:tpm     → token-bucket counter + ts     # rate limiting
thread:<id>:cache        → serialized recent messages     # hot-thread cache
idem:<key>               → stored response / in-flight     # idempotency
queue:inference          → LPUSH/BRPOP job list           # job queue
stream:<reqid>           → PUBLISH / SUBSCRIBE channel     # token fan-out

That last one is the trick that makes streaming scale: the model worker PUBLISHes each token to stream:<reqid>, and whichever gateway instance is holding the client’s SSE connection is SUBSCRIBEd — so the worker and the connection-holder need not be the same machine.

Object storage (S3-like) — big blobs. Attachments, images, uploaded documents, exported transcripts. Cheap, durable, infinitely scalable, and serve-able via signed URLs straight from the CDN. Postgres stores a pointer (the object key); the bytes never touch the database.

Vector DB (optional) — retrieval & memory. Embeddings of past threads, uploaded docs, or a knowledge base, indexed for nearest-neighbor search. When a thread should “remember” earlier context that has scrolled out of the window, or answer over a document, you embed the query and pull the top-k relevant chunks into the prompt (RAG).

Analytics / warehouse (OLAP) — offline truth. A columnar store for events, evals, and dashboards. You never run product analytics on the operational Postgres — long scans there would hurt the live path. Usage and interaction events stream into the warehouse for cost dashboards, quality evals, and product metrics.

Pick the store for the property, not the habit

Durability and relations → Postgres. Speed and ephemerality → Redis. Large opaque bytes → object storage. Similarity search → vector DB. Big offline scans → warehouse. The interview answer is never “put it all in Postgres” or “cache everything in Redis” — it is naming the one property that forces each choice.

6 · Inference layer

The orchestration service does not talk to GPUs directly. It hands the job to an inference gateway, which fronts a scheduler/router, which dispatches to an autoscaled fleet of GPU model servers. The cleverness lives in the scheduler and the servers:

  • Continuous (in-flight) batching. Rather than wait to assemble a fixed batch, the server adds and removes sequences from the running batch every step, so a new request joins almost immediately and finished ones leave — keeping the GPU saturated without making anyone wait for a batch to fill.
  • Paged KV-cache. Each request’s growing KV cache is stored in fixed-size pages (like virtual memory) instead of one contiguous block, so memory isn’t fragmented and you can pack far more concurrent sequences onto a card.
  • Prefix / prompt caching. The server keeps the computed KV state of stable prefixes (system prompt, early history). A repeat request with the same prefix is a cache hit — that prefill is skipped entirely, slashing TTFT and cost (cached input often bills at ~0.1×).
  • Admission control / backpressure. When the fleet is saturated, the scheduler refuses or queues new work rather than thrashing — an explicit signal upstream instead of a meltdown.
  • Model-tier routing & fallback. Route by task: a small/fast tier (Haiku 4.5) for easy turns, balanced (Sonnet 4.6) for the everyday, most-capable (Opus 4.8) for hard reasoning — with a fallback tier if the primary is down.

7 · Cross-cutting concerns

Wrapping every box: observability (distributed traces across the hops, metrics for p50/p99 latency, TTFT, tokens/sec, error rate, and cost — sliced per tenant; structured logs with the request id); secrets management (keys and DB creds from a vault, never in code); feature flags (ship a new model tier to 1% first); autoscaling (stateless services scale on CPU/RPS, the GPU fleet on queue depth); multi-region (data residency + failover); and reliability (timeouts on every outbound call, retries with budgets and jitter, idempotency keys, circuit breakers when a dependency is clearly down, and graceful degradation — a smaller model beats an error).

Part B · Trace one request: “send a message in a thread”

Now the payoff: one request, numbered hop by hop, from the browser to the attention math and back. Follow the stepper, then read the concrete examples under each stage.

Step 1 · The client sends the message

The browser opens an HTTPS request. stream:true asks for an SSE response; the Idempotency-Key is a fresh UUID the client generates per send so a retry is safe.

POST /v1/threads/3f9a.../messages HTTP/2
Host: api.example.com
Authorization: Bearer sess_8e21c0...
Idempotency-Key: 5b1e-44a2-9c7f-01
Content-Type: application/json
Accept: text/event-stream

{ "content": "Summarize the thread so far in 3 bullets.", "stream": true }

Step 2 · Edge → gateway

The edge terminates TLS, runs WAF rules, and geo-routes to the nearest region’s load balancer. The gateway then does its checklist against Redis — all sub-millisecond:

# auth: resolve the session token → user + tenant
GET session:sess_8e21c0...        → {user_id, tenant_id, exp}   # miss → 401

# idempotency: claim the key; if it already exists, replay the stored result
SETNX idem:5b1e-44a2-9c7f-01 "in-flight"  EX 86400

# rate limit: token bucket, BOTH dimensions, atomically (Lua)
EVAL token_bucket  ratelimit:<user>:rpm  1            # 1 request
EVAL token_bucket  ratelimit:<user>:tpm  <est_tokens>  # estimated cost

Input moderation starts in parallel. If the bucket is empty, the gateway short-circuits before any model work:

HTTP/2 429 Too Many Requests
Retry-After: 7
Content-Type: application/json

{ "error": { "type": "rate_limit", "message": "Token budget exceeded; retry after 7s." } }

Step 3 · Conversation service: ACL, load history, append

First the deep ACL check — does this user own this thread? Then load history: try the hot cache, fall back to Postgres.

# fast path: hot-thread cache
GET thread:3f9a...:cache          # hit → recent messages, skip the DB

# slow path on a miss: the indexed read
SELECT role, content, created_at
FROM messages
WHERE thread_id = $1
ORDER BY created_at ASC
LIMIT 200;                       # uses idx_messages_thread_time

Append the new user message, then invalidate (or update) the cache so the next read is consistent:

INSERT INTO messages (thread_id, role, content, input_tokens)
VALUES ($1, 'user', $2, $3)
RETURNING id, created_at;

# then in Redis
DEL thread:3f9a...:cache          # invalidate; or RPUSH the new message + EXPIRE

Step 4 · Orchestration assembles the prompt

The orchestration service builds the model input: the system prompt, then the prior turns, then the new message. It places a prompt-cache breakpoint on the stable prefix — the system prompt and early history — so on repeat turns that prefix is a cache read (~0.1×) instead of a fresh prefill. It counts tokens, picks the tier, enqueues the job, and subscribes to the stream.

messages = [
  { "role": "system",
    "content": [ { "type": "text", "text": "<long stable system prompt>",
                  "cache_control": { "type": "ephemeral" } } ] },  # ← breakpoint: cache the prefix
  { "role": "user",      "content": "<turn 1>" },
  { "role": "assistant", "content": "<reply 1>" },
  { "role": "user",      "content": "Summarize the thread so far in 3 bullets." }  # ← dynamic tail
]
LPUSH queue:inference  {reqid, tenant, model:"sonnet-4-6", messages, max_tokens:512}
SUBSCRIBE stream:<reqid>          # this gateway now relays tokens as they arrive
Cache the head, vary the tail

Stable content first (system prompt, tools, early history), volatile content last (the new user turn). That ordering is what makes the prefix a byte-identical cache hit every turn. Put a timestamp or the user’s name at the top and you invalidate the cache for everything after it — re-paying full prefill on every message.

Step 5 · Inference gateway → scheduler

A worker BRPOPs the job. The inference gateway asks the scheduler for a model server with capacity (admission control — if none, the job waits in the bounded queue, applying backpressure). The chosen server checks its prefix/KV cache: a prompt-cache HIT on the stable prefix means prefill is skipped for those tokens, and only the dynamic tail is prefilled fresh.

Step 6 · The model-server forward pass (attention + MLP)

This is the chef finally cooking. The server turns text into tokens, tokens into vectors, and runs them through L identical transformer layers — each layer is self-attention (the social step) followed by an MLP (the private step), with residual adds around both. Watch the two phases.

In pseudocode, the prefill processes the whole prompt at once and fills the cache:

def prefill(prompt_ids, cache):
    x = embed(prompt_ids)                 # [B, S, H]  residual stream
    for layer in layers:                  # L transformer blocks
        q, k, v = layer.qkv(x)            # project: [B, heads, S, head_dim]
        cache[layer].store(k, v)          # WRITE K/V for every prompt token
        a = sdpa(q, k, v, is_causal=True) # causal attention, all positions at once
        x = x + layer.o(a)                # attention + residual
        x = x + layer.mlp(x)              # gelu(expand)→contract + residual
    logits = unembed(x[:, -1])            # only the last position matters now
    return sample(logits)                 # the FIRST output token → TTFT

Then the decode loop runs one token at a time, computing K/V only for the single new token and attending over everything already cached:

def decode_step(prev_token, cache):
    x = embed(prev_token)                 # [B, 1, H]  just ONE token
    for layer in layers:
        q, k, v = layer.qkv(x)            # Q/K/V for the one new token
        k_all, v_all = cache[layer].append(k, v)  # APPEND, get full history
        a = sdpa(q, k_all, v_all)         # attend over ALL cached K/V
        x = x + layer.o(a)
        x = x + layer.mlp(x)
    return sample(unembed(x[:, -1]))      # next token; loop until EOS / max_tokens
ELI5

Prefill is the chef reading the entire recipe and prepping every ingredient on the counter in one burst — fast, all at once. Decode is plating the dish one spoonful at a time, glancing at the whole prepped counter (the KV cache) before each spoonful, and never re-prepping what’s already laid out. The reading is quick; the one-spoonful-at-a-time plating is why a long answer takes a while.

Step 7 · Streaming back

Each generated token is PUBLISHed to stream:<reqid>. The gateway instance holding the client’s SSE connection is SUBSCRIBEd and forwards each as an SSE data: event. Output moderation runs incrementally; on a flag it can cut the stream mid-flight.

HTTP/2 200 OK
Content-Type: text/event-stream

event: message_start
data: {"id":"msg_a1...","model":"sonnet-4-6"}

event: content_block_delta
data: {"delta":{"text":"- The thread"}}

event: content_block_delta
data: {"delta":{"text":" covers the Q3 plan"}}

event: message_stop
data: {"usage":{"input_tokens":1840,"output_tokens":96,"cache_read_tokens":1600}}

Step 8 · Completion

When the stream ends the work becomes durable. Persist the assistant message, record usage (separating cache_read vs cache_creation tokens, since they bill differently), refresh the caches with a TTL, maybe enqueue title generation for a new thread, and emit analytics events to the warehouse.

INSERT INTO messages (thread_id, role, content, input_tokens, output_tokens)
VALUES ($1, 'assistant', $2, 1840, 96);

UPDATE threads SET updated_at = now() WHERE id = $1;
RPUSH thread:3f9a...:cache <assistant msg>   EXPIRE thread:3f9a...:cache 3600
SET   idem:5b1e-44a2-9c7f-01 <final result> EX 86400   # replay on retry
LPUSH queue:billing  {tenant, model, input:1840, output:96, cache_read:1600}
LPUSH queue:titlegen {thread_id}             # only if title is still null

Step 9 · Failure modes at each hop

Gotcha: the stateful streaming connection

The SSE connection lives on one gateway instance, but the model worker that produces tokens is a different machine. If you try to have the worker write directly to the socket, you couple them and break horizontal scaling. The fix is the indirection in step 7: the worker PUBLISHes to Redis, and whichever instance holds the socket SUBSCRIBEs. Keep the connection-holder dumb and stateless; let the shared store do the fan-out. The same indirection is what makes reconnect-and-resume possible.

How this shows up in the interview

Q1 · “Design the chat API with streaming and threads.”

Start with the resource model and the streaming contract. “Threads own messages; a send is POST /v1/threads/{id}/messages with stream:true, and the reply comes back as SSE.” Then walk the spine: stateless services behind a gateway, Postgres as the system of record, Redis for sessions/limits/cache and a pub-sub stream for token fan-out, and an inference gateway in front of the GPU fleet. Make three things explicit: optimistic UI + reconnection on the client (resume via Last-Event-ID), idempotency keys so a retried send doesn’t double-post, and the pub/sub indirection that lets any gateway instance relay tokens from any worker. That last point is what separates “I’ve called a streaming API” from “I’ve built one.”

Q2 · “What is each data store for, and why that store?”

Name the property that forces each choice. Postgres — durability + relations + transactions, so messages are never lost and ACLs join cleanly. Redis — sub-ms shared state for sessions, rate-limit counters, the hot-thread cache, idempotency keys, the job queue, and pub/sub streaming; it’s ephemeral on purpose. Object storage — large opaque blobs (attachments, exports) served from the CDN, with only a pointer in Postgres. Vector DB — nearest-neighbor search for retrieval/memory when relevant context has scrolled out of the window. Warehouse — offline columnar scans for evals and cost dashboards, kept off the live Postgres. The wrong answer is one store doing all five jobs.

Q3 · “Where does the latency go, and how do you cut it?”

Separate the two clocks. TTFT = queue wait + prefill, so it’s driven by prompt length and whether the stable prefix is a prompt-cache hit — cache the prefix and TTFT collapses. Total time is dominated by decode: each output token is its own sequential forward pass reading the whole KV cache, so the lever is a shorter output (cap max_tokens, instruct for brevity) or a faster tier. Platform overhead (network, auth, DB, queue) is a sliver — don’t micro-optimize it before the model. And always stream, so perceived latency tracks TTFT, not the full decode. Report p99, not the average.

Q4 · “Trace a request end to end.”

Walk Part B out loud: client POST with bearer token + idempotency key → edge (TLS/WAF/geo) → gateway (auth via Redis, idempotency SETNX, token-bucket rate limit on requests and tokens) → conversation service (ACL, load history from cache-or-Postgres, INSERT the message) → orchestration (assemble system+history+turn, set the prompt-cache breakpoint, enqueue, subscribe) → scheduler admits to a GPU server → forward pass (tokenize, embed, prefill all tokens through attention+MLP ×L filling the KV cache, sample the first token = TTFT, then the decode loop appending K/V per token) → each token PUBLISHed to Redis and forwarded as SSE → completion (persist the assistant message, meter cache_read vs cache_creation, refresh caches, emit analytics). The signal you’re sending: cheap rejections high and early, the expensive math deep and late, tokens streaming back up through pub/sub.

Check your understanding

Why route token streaming through Redis pub/sub instead of having the GPU worker write straight to the client’s connection?

Because the SSE connection lives on one stateless gateway instance while the worker that generates tokens is a different machine, and the scheduler may even move the work between servers. If the worker wrote directly to the socket you’d couple producer and connection-holder, breaking horizontal scaling and making reconnects impossible. Publishing each token to stream:<reqid> and letting whichever instance holds the socket subscribe decouples them: any worker can feed any connection, and a reconnecting client can resume from the stream offset.

In the forward pass, why is the first output token (TTFT) produced by a single parallel pass, but the next 200 tokens take far longer in total?

The first token comes out of prefill: all prompt tokens go through the L layers together in one compute-bound, parallel pass that also fills the KV cache — fast per token. Every token after that is a decode step: a separate sequential forward pass that reads all the weights plus the entire growing KV cache out of memory to produce just one token, then appends its K/V and loops. Decode is memory-bandwidth-bound and strictly serial (token N needs token N-1), so 200 output tokens are 200 back-to-back passes — which is why decode, not prefill, dominates total latency.

A client’s phone drops the connection, retries the send, and the user ends up with two identical messages and a double bill. What was missing, and where does it get fixed?

The idempotency mechanism. The client should send a stable Idempotency-Key per logical send, and the gateway should claim it (SETNX idem:<key>) before doing any work. On the retry the key already exists, so the gateway replays the stored result instead of inserting a second message and re-running generation. It’s fixed at the gateway (step 2), reinforced by the conversation service treating the insert as part of the idempotent unit, and the stored final result (step 8) is what gets replayed. Pair it with reconnect-via-Last-Event-ID so the resumed client re-attaches to the original stream rather than starting a new one.
