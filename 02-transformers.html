Module 02

How a transformer computes

The whole course rests on one distinction — weights vs. activations. To really feel it, you need to see what a transformer actually does to your input, end to end, and watch both kinds of numbers appear. No prior ML needed; if you can read a for-loop and a matrix multiply, you're ready.

The one-sentence job of a transformer

A language transformer does exactly one thing: given a sequence of tokens, predict the next token. Everything else — chat, code, summarization — is that single operation run over and over. "The cat sat on the ___" → "mat". That's it. The entire architecture below exists to compute that prediction well.

A mental model you already have

Think of the model as a giant pure function: predict(tokens) → next_token_probabilities. The weights are the millions of constants baked into that function (like a huge config file or lookup table loaded at startup). The activations are the local variables that exist only while one call is running — request-scoped scratch memory. Same split as a web server: the compiled code + loaded config is fixed; the per-request data comes and goes. Hold that analogy; it is the weights/activations distinction.

Step 0 — Tokens: text becomes integers

Text is first chopped into tokens (roughly word-pieces) and each is mapped to an integer ID. "The cat sat" might become [464, 3797, 3332]. A sequence of S tokens is just a list of S integers. If you process several sequences at once, that's the batch B. So the raw input is a grid of integers of shape [B, S]. Nothing learned yet — just indexing.

Step 1 — Embedding: integers become vectors

A neural network can't do math on the integer "3797". So we look each token ID up in a big table that maps every possible token to a vector of H numbers. That table is the embedding matrix E, shaped [vocab_size, H] — one row per possible token.

vocab_size = 50000,  H = 4096
E: [50000, 4096]          # the lookup table — a WEIGHT (learned, fixed)

token id 3797  ->  E[3797]  ->  a vector of 4096 numbers

Do that for every token in the input and you get your first activation tensor:

X = E[token_ids]          # shape [B, S, H] — an ACTIVATION (depends on the input)
Your first weight and your first activation, side by side

E (the [50000, 4096] table) is a weight: it's the same for every request, learned once during training, and its size depends only on the vocabulary and H — not on your input. X (the [B, S, 4096] result) is an activation: it only exists because you fed in a specific batch, and it grows when you add more tokens or more sequences. This is the whole distinction, in miniature.

Step 2 — The residual stream: the data bus through the model

From here on, that [B, S, H] tensor is the only thing flowing through the network. Every token is now a point in H-dimensional space (here H = 4096). The model's job is to nudge each token's vector, layer by layer, until it encodes enough context to predict what comes next. This flowing tensor is called the residual stream.

"Residual" because each layer doesn't replace the stream — it computes an update and adds it on:

X = X + attention(X)      # add what attention learned
X = X + mlp(X)            # add what the MLP learned

Think of it as a shared mutable buffer that each layer reads from and writes a correction back into. The shape [B, S, H] never changes as it passes through — only the values do.

Step 3 — A transformer is just N identical blocks stacked

The model body is a loop. You apply the same kind of block L times (the test's reference model has L = 32 layers). Each block has its own private copy of the weights, but the structure is identical:

X = embed(token_ids)              # [B, S, H]
for layer in range(L):           # L = 32 blocks
    X = X + attention_layer(X)    # the "social" step (module 3)
    X = X + mlp_layer(X)          # the "private" step
logits = X @ E.T                  # project back to vocab -> next-token scores

So the famous "transformer" is, structurally, an embedding lookup, then 32 repetitions of {attention, then MLP}, then a projection back out to vocabulary scores. Two matrix-heavy sub-layers per block. Module 3 unpacks attention; the MLP we'll do right here because it's the cleanest place to count weights and activations.

Step 4 — The MLP, computed by hand

The MLP (feed-forward network) is two matrix multiplies with a nonlinearity between them. It processes each token independently. Here are its weights:

H = 4096,  I = 4 * H = 16384      # I = intermediate ("expansion") size

W_in:  [I, H] = [16384, 4096]     # WEIGHT — expands the width
W_out: [H, I] = [4096, 16384]     # WEIGHT — contracts it back

And here's the computation on the residual stream X: [B, S, H]. Watch the shapes — and notice the new activation that appears in the middle:

hidden = gelu(X @ W_in.T)    # [B,S,4096] @ [4096,16384] -> [B, S, 16384]   ACTIVATION
out    = hidden @ W_out.T    # [B,S,16384] @ [16384,4096] -> [B, S, 4096]   ACTIVATION

(Recall from the matrix convention we'll formalize in module 3: F.linear(X, W) computes X @ W.T, and weights are stored output-first as [out, in].)

Read the two kinds of numbers in this block

Weights here: W_in and W_out. Their sizes — 16384×4096 each — depend only on H and I. Feed in 10 tokens or 10 million; these matrices are byte-for-byte identical. Activations here: hidden and out. Their sizes — [B, S, …] — scale directly with how many tokens you pushed through. Double the sequence, double these tensors. This is why long context blows up activation memory but not weight memory.

Counting it out: where the memory actually goes

Abstract until you put numbers on it. Let's use the test's reference scale: H = 4096, I = 16384, L = 32 layers. (Numbers are in parameters/elements; multiply by 2 bytes for bf16, or 4 for fp32, to get bytes.)

Weights (model state) — fixed, independent of input

Weight (per layer)ShapeParameters
W_q, W_k, W_v, W_o (attention)4 × [4096, 4096]~67 M
W_in, W_out (MLP)2 × [16384, 4096]~134 M
Per layer total—~201 M
× 32 layers—~6.4 B parameters

Notice what's absent from this table: B and S. The parameter count has nothing to do with your input length. A 6.4-billion-parameter model is 6.4B parameters whether you prompt it with 5 tokens or 50,000. That's model-state memory — and it's what Tensor Parallelism (module 5) shrinks by slicing these matrices across GPUs.

Activations — scale with batch and sequence

Now the other axis. Just the MLP's hidden tensor, for one layer, at the test's B=1:

hidden: [B, S, I] = [1, S, 16384]
  S =  8192  ->  ~134 M elements   (~0.27 GB in bf16)
  S = 65536  ->  ~1.07 B elements  (~2.1 GB in bf16)   # just ONE tensor, ONE layer!

Eight times the tokens, eight times the activation memory — and that's a single intermediate tensor. Add up Q, K, V, attention scores, hidden, and the residual stream across all layers and the activation footprint becomes enormous at long context. That's activation memory — and it's what Sequence Parallelism (module 6) shrinks by splitting the S axis across GPUs.

The punchline that sets up the whole course

Two memory bills, two completely different drivers. Weights are driven by model size (H, I, L, vocab) and are constant per request. Activations are driven by workload size (B, S) and vanish when the request ends. You can be crushed by either one independently — and each parallelism strategy in this course targets one or both. Keep this table in your head; module 8 is literally measuring these two quantities on real GPUs.

One token's journey (the whole thing in plain words)

  1. The word "cat" → token id 3797 (just an integer).
  2. Look it up in embedding table E → a 4096-number vector. It joins the residual stream X.
  3. Layer 1 attention: "cat" looks at the earlier tokens, gathers context ("the", subject position…), and adds an update to its vector.
  4. Layer 1 MLP: "cat"'s vector is transformed on its own and an update is added.
  5. Repeat for 32 layers — each one refines the vector with more context.
  6. Final vector → multiply by E.T → a score for every possible next token → highest score wins ("sat", maybe).

Every "look up", "add an update", and "transform" above is a matrix multiply between an activation (the stream) and a weight (a learned matrix). That's the entire machine. Now you're ready to open up the most interesting sub-layer — attention — in module 3.

Check your understanding

Classify each as a weight or an activation: (a) the embedding table E, (b) the residual stream X, (c) W_in, (d) the MLP's hidden tensor.

(a) E — weight. Learned lookup table; size depends on vocab and H, not on input. (b) X — activation. Exists only for a specific batch; shape [B,S,H] scales with input. (c) W_in — weight. Learned [I,H] matrix, constant per request. (d) hidden — activation. Produced during a forward pass; shape [B,S,I] scales with input. Rule of thumb: if its shape contains B or S, it's an activation.

You double the sequence length from 8K to 16K tokens. Which grows — the parameter count, the activation memory, both, or neither? Why?

Only activation memory grows (it roughly doubles). The parameter count is a property of the matrices' shapes (H, I, L, vocab) and is completely independent of how many tokens you process. Activations carry a S dimension, so they scale directly with sequence length. This asymmetry is exactly why a model whose weights fit fine can still OOM on long inputs.

What does "residual stream" mean, and why does the tensor keep the shape [B, S, H] through all 32 layers?

The residual stream is the running [B,S,H] tensor that flows through the model — each token's evolving vector. Layers don't replace it; they compute an update and add it back (X = X + sublayer(X)). Because every sub-layer maps [B,S,H] to a same-shaped update that's added on, the stream's shape is invariant — only its values change. That's also why you can stack identical blocks: each one takes and returns the same shape.

In the MLP, W_in is [16384, 4096] and hidden is [B, S, 16384]. If a GPU is short on memory, which of these could you shrink by adding more GPUs, and which technique targets which?

Both, but via different strategies. W_in is a weight — slice it across GPUs with Tensor Parallelism (each GPU holds part of the [16384, 4096] matrix). hidden is an activation with an S axis — shrink it with Sequence Parallelism (each GPU processes only some of the tokens, so its hidden is [B, S/D, 16384]). TSP does both at once. You've just previewed modules 5–7.
