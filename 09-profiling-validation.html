Module 09

Profiling & validation

A parallel implementation is worthless if it's either wrong or unmeasured. This module covers the two things the test does around the algorithms: prove correctness, and measure cost.

Part 1 — Proving correctness with a reference oracle

How do you know your 8-GPU attention computes the same thing as plain attention? You compare against a single-GPU reference — an "oracle" you trust because it's simple. This is the same instinct as testing an optimized function against a naive one.

The recipe (from report.md and the runners):

  1. On rank 0, build the full weights and input, and compute the plain single-device output: expected = attn(X, W_q, W_k, W_v, W_o).
  2. broadcast those same weights/input to all ranks, so everyone starts identical.
  3. Run the parallel implementation to get out.
  4. Assert out matches the corresponding slice of expected, within a tolerance.
  5. Optionally all_gather every rank's output and check the reassembled full tensor equals the full reference.
expected = attn(X, W_q, W_k, W_v, W_o, num_heads)   # trusted single-GPU answer
out = tp_attn(X, W_q_p, W_k_p, W_v_p, W_o_p, p_heads)
torch.testing.assert_close(out, expected, rtol=1e-5, atol=1e-5)
Why a tolerance, not exact equality?

Floating-point addition isn't associative — summing numbers in a different order gives slightly different results. The parallel version sums partials in a different order than the reference, so outputs differ in the last few bits. atol/rtol = 1e-5 says "equal up to expected floating-point noise." Demanding exact equality would fail on correct code.

Two refinements the sessions arrived at, worth absorbing:

  • Validation is expensive and distorts profiling. Computing the full dense reference on every rank costs more than the implementation itself. So the mature workflow is: validate on small shapes, then profile big shapes separately with validation turned off. The repo went back and forth adding/removing these checks for exactly this reason.
  • Warm up before timing. The first run includes one-time CUDA setup. The runners do an untimed warmup call, then torch.cuda.synchronize(), then profile a second call.

Part 2 — Measuring memory and latency

Two numbers matter for this kind of work:

  • Peak memory per GPU — the high-water mark of allocated memory. This is what decides whether the model fits. torch.cuda.max_memory_allocated() reports it.
  • Latency — wall-clock time for the region of interest. In a distributed run, the number that matters is the slowest rank (everyone waits for it), reported as max_rank_elapsed_ms. Mean-rank time is also tracked, to reveal imbalance.

The repo wraps each attention call in a profiler context (profile_rank_region in profiler_utils.py) that does, per rank:

dist.barrier()                          # line up all ranks
torch.cuda.reset_peak_memory_stats()    # zero the high-water mark
with torch.profiler.profile(
        activities=[CPU, CUDA],
        profile_memory=True,
        record_shapes=True) as prof:
    ... run the attention ...
torch.cuda.max_memory_allocated()       # peak this rank hit
prof.export_chrome_trace(...)           # per-rank trace file you can open in a viewer
Per-rank, not global

Each rank writes its own trace file (e.g. tp_attention_rank_3_trace.json). There's no single global profiler — you collect 8 files and look at the worst one. The repo even stores them in a shared Modal Volume so all ranks' traces land in one place after the run.

Where it runs: Modal + torchrun

You don't own 8 H100s, so the test rents them on-demand via Modal (a serverless GPU platform). The chain, from iterate.py:

  1. Modal provisions a container with h100:8 — eight H100 GPUs.
  2. Inside, it runs torchrun --nproc_per_node=8 tensor_parallelism.py — spawning the 8 ranks from module 4.
  3. Each rank binds to its LOCAL_RANK GPU, runs, and writes its trace to a mounted Volume.
  4. A guard checks world_size == EXPECTED_WORLD_SIZE so a misconfigured launch fails loudly instead of silently using fewer GPUs.
uv run modal run iterate.py        # runs tp, then sp, then tsp
uv run modal run iterate.py::ts    # run just TSP
Real-world gremlins the sessions hit

Distributed GPU runs fail in ways local code never does: a NCCL NVLink/Fabric-Manager multicast error (fixed by setting NCCL_NVLS_ENABLE=0); workers interrupted during import torch when three heavy jobs ran back-to-back (fixed by running TSP as its own job); and a missing matplotlib breaking the memory-timeline export. None are bugs in the algorithm — they're the texture of operating real multi-GPU infrastructure. Expect this layer to need its own debugging.

Reading the output

A run prints lines like:

tp_attention max_rank_elapsed_ms=130.813
peak_allocated=0.601562 GiB
tensor-parallel causal attention completed

That's one row of the tables you read in module 8: a mode, a latency (slowest rank), and a peak memory. Sweep over sequence lengths and head counts and you've reproduced the paper's measurements — which is the whole deliverable of the test.

Check your understanding

Why compare the parallel output to a single-GPU reference instead of just trusting it because "it ran"?

Because a parallel implementation can run to completion and still compute the wrong answer — a mis-sharded weight, a wrong collective, a bad causal offset. The single-GPU attn is simple enough to trust, so it serves as an oracle: if the parallel output matches it (within float tolerance), the sharding and collectives are correct. "It ran without crashing" proves nothing about correctness.

Why is the reported latency the MAX across ranks, not the mean?

Because the ranks run in lockstep through collectives — the whole step isn't done until the slowest rank finishes. End-to-end wall time is governed by the max-rank time. The mean is still tracked, but only to reveal load imbalance (e.g. the causal-imbalance problem from module 6); it's not the latency you'd actually experience.

Why does the mature workflow separate correctness checks from profiling runs?

Because computing the full dense reference attention on every rank is more expensive than the parallel implementation being measured — so leaving it in distorts the timing and inflates memory. You validate once on small shapes (cheap, decisive), then profile the real large shapes with validation disabled so the numbers reflect only the implementation. The repo's add/remove churn around the correctness checks was exactly this realization in motion.
