Tensor-Sequence Parallelism,
explained for software engineers

You're a full-stack engineer, not a data scientist. This course rebuilds every idea in the mirendil test from the ground up — slowly, concretely, and shape-first — until TP, SP, and TSP feel obvious.

What this course is about

The test is a reproduction study of a research paper on Tensor-Sequence Parallelism (TSP). The goal: run a transformer's attention and MLP across 8 GPUs so that no single GPU runs out of memory, and measure whether the clever combined strategy (TSP) really beats the simpler ones (TP, SP).

Every module ends with check-your-understanding toggles. There's an interactive shape calculator in modules 5–7 so you can play with the tensor shapes yourself.

The path

MODULE 01
The big picture

Why we split models across GPUs at all. The two kinds of memory pressure — weights vs. activations — and the three strategies that attack them.

 MODULE 02
How a transformer computes

Start here. What a transformer actually does to your text, step by step — embeddings, the residual stream, layers — and exactly which numbers are weights and which are activations. The foundation for everything else.

 MODULE 03
Attention & the transformer block

A no-heavy-math refresher: B, S, H, heads, head_dim, the Q/K/V/O projections, the MLP, and the one matrix convention you must internalize.

 MODULE 04
Multi-GPU primitives

torchrun, ranks, and the three collective operations — broadcast, all_reduce, all_gather — that everything else is built from.

 MODULE 05
Tensor parallelism (TP)

Sharding attention heads. Column-parallel vs row-parallel. Why the output projection needs an all_reduce. The insight people always trip on.

 MODULE 06
Sequence parallelism (SP)

Sharding tokens instead of heads. Why attention suddenly needs to gather K/V from everyone. Causal load imbalance and the zigzag fix.

 MODULE 07
Tensor-sequence parallelism (TSP)

Folding both axes onto one set of GPUs. The "diagonal grid" problem, the loop over weight shards, and Algorithm 1 line by line.

 MODULE 08
Tradeoffs & when to use what

Memory scaling (1/T, 1/Σ, 1/D), communication cost, and the honest answer to "is TSP just better?" — read against the real benchmark numbers.

 MODULE 09
Profiling & validation

How you actually measure peak memory and latency on 8×H100, and the reference-output trick that proves a parallel implementation is correct.

 MODULE 10
Glossary & cheat sheet

Every symbol, every collective, every shape in one place. Keep this open while you read the actual code in the repo.

How to use this

Read in order — each module assumes the previous one. When a module references a file like tensor_parallelism.py, that's real code sitting next to this course in the repo; open it side by side once you've read the concept. You don't need a GPU to follow along. The notebooks above let you actually run and tinker with each concept (multi-GPU is simulated on CPU).
