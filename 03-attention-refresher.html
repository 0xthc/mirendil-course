Module 03

Attention & the transformer block

You can't parallelize something you can't picture. This module gives you exactly enough of attention to follow the rest — focused on shapes and data flow, not the theory of why attention works.

The vocabulary of shapes

Almost every confusion in this whole topic is really a shape confusion. Learn these five letters and you're 80% of the way there. They appear constantly in the code and the paper.

SymbolNameMeaningExample value in the test
Bbatch sizeHow many independent sequences we process at once.1 or 8
Ssequence lengthHow many tokens are in each sequence.8192 → 65536
Hhidden sizeThe width of each token's vector. Also called the model dimension / embedding dim / residual width.2048 or 4096
num_headsnumber of headsAttention is split into this many parallel "heads".16 or 32
head_dim (D)head dimensionThe width each head works in. Always H / num_heads.128
Heads-up on overloaded letters

The letter D is unfortunately used for two different things across the paper and code. In attention, D often means head_dim. In the parallelism discussion, D means the number of GPUs / parallel degree. Always check the context. This course writes head_dim when it means the head width, and D for the GPU count.

A token is a vector of width H

Your input X has shape [B, S, H]: a batch of B sequences, each S tokens long, each token being a vector of H numbers. That single tensor is the thing flowing through the whole network. Everything below transforms it and hands back something the same shape.

The one matrix convention you must internalize

In math textbooks you often see y = W·x (weight on the left, column vectors). PyTorch and this code use the opposite, row-vector convention:

Y = X @ W.T        # what F.linear(X, W) actually computes

Concretely, F.linear(X, W) does X @ W.transpose(). So PyTorch stores a weight as [out_features, in_features] — output size first. This trips up everyone coming from a math background. Burn it in now:

Remember

F.linear(X, W) with X: [B, S, in] and W: [out, in] produces [B, S, out]. The weight is stored output-first, and it gets transposed for you. When module 5 talks about slicing a weight's "rows" vs "columns", it means rows/columns of this [out, in] matrix.

ELI5

Each word raises its hand and asks: "who here is relevant to me?" Every other word answers, and the asking word listens more to the relevant answers and less to the rest — then updates its own understanding. "It" might listen hard to "cat" and ignore "the". Do that for every word at once and you have attention. Causal just means you may only ask words that came before you — no peeking at the future.

Attention, as a data-flow pipeline

Forget the intuition about "tokens attending to each other" for a second and just trace the tensors. Here is the real reference implementation from tensor_parallelism.py, lightly annotated:

def attn(X, W_q, W_k, W_v, W_o, num_heads):
    B, S, H = X.shape
    head_dim = H // num_heads

    # 1. Project X into three new tensors: Query, Key, Value.
    #    Each projection is just a matrix multiply. q,k,v: [B, num_heads, S, head_dim]
    q = project_to_heads(F.linear(X, W_q), num_heads, head_dim)
    k = project_to_heads(F.linear(X, W_k), num_heads, head_dim)
    v = project_to_heads(F.linear(X, W_v), num_heads, head_dim)

    # 2. The attention operation itself. Causal = each token only sees
    #    itself and earlier tokens. attn: [B, num_heads, S, head_dim]
    attn = F.scaled_dot_product_attention(q, k, v, is_causal=True)

    # 3. Glue the heads back together: [B, num_heads, S, head_dim] -> [B, S, H]
    attn = attn.transpose(1, 2).contiguous().view(B, S, H)

    # 4. One more projection (the "output projection"). out: [B, S, H]
    return F.linear(attn, W_o)

Four steps. Notice the input was [B, S, H] and the output is [B, S, H] — same shape. Let's name what each step is for.

Step 1 — Q, K, V projections

Three separate weight matrices (W_q, W_k, W_v, each [H, H]) turn each token into three new vectors: a Query ("what am I looking for?"), a Key ("what do I offer?"), and a Value ("what information do I carry?"). Then we reshape [B, S, H] into [B, num_heads, S, head_dim] — we slice the width H into num_heads independent lanes. This slicing is exactly what TP will exploit.

A real question from the original sessions

"Why is Q/K/V in [B, heads, S, head_dim] and not [B, S, heads, head_dim]?" — Because PyTorch's scaled_dot_product_attention (SDPA) expects the head axis right after the batch axis. You build it as [B, S, heads, head_dim] then transpose(1, 2) to swap S and heads. The heads become a "batch-like" dimension that SDPA processes in parallel.

Step 2 — scaled dot-product attention (causal)

This is the actual "attention". For each head, every query token compares itself against every key token, and uses those scores to take a weighted blend of the value vectors. Causal means a token at position i may only look at positions 0…i (the past), never the future. is_causal=True applies that mask for you. The crucial fact for later: a query for token i needs the keys/values of all tokens up to i. Hold that thought — it's why SP is hard.

Step 3 — concatenate heads

Each head produced its own head_dim-wide output. We stitch them back side by side into a full H-wide vector per token. transpose(1,2).view(B, S, H) does the stitching.

Step 4 — the output projection W_o

One final matrix W_o: [H, H] mixes the concatenated head outputs back into the token's residual stream. This little step has a subtle role that module 5 spends real time on, because it's where most people's mental model of TP breaks. Plant a flag here.

The MLP block (the easy sibling)

After attention, each transformer block has a feed-forward network — the MLP. It's mercifully simple: two matrix multiplies with a nonlinearity between them, applied to each token independently.

def mlp(X, W_in, W_out):
    hidden = F.gelu(F.linear(X, W_in))   # [B, S, H] -> [B, S, I]   (I = 4*H, "expansion")
    return F.linear(hidden, W_out)        # [B, S, I] -> [B, S, H]

The first matrix expands the width from H to an intermediate size I (typically 4·H), applies GELU, then the second matrix contracts back to H. The key property: the MLP touches each token on its own. Token 5's output depends only on token 5's input. There's no cross-token mixing here (all the cross-token mixing happened in attention). This is why, as you'll see, the MLP is trivially easy to shard by tokens — no communication needed.

Mental model for the whole block

Attention = the social step (tokens look at each other, needs coordination). MLP = the private step (each token thinks on its own, embarrassingly parallel). Every parallelism strategy in this course is easy on the MLP and interesting on attention — because attention is the only place GPUs are forced to talk.

Check your understanding

If H = 4096 and num_heads = 32, what is head_dim? And what shape is one Q tensor before SDPA?

head_dim = H / num_heads = 4096 / 32 = 128. The Q tensor is [B, 32, S, 128] — batch, heads, sequence, head_dim. The 32 × 128 = 4096 exactly reconstructs H when you concatenate the heads back together.

You see W_q stored with shape [4096, 4096] and call F.linear(X, W_q) where X is [B, S, 4096]. What's the output shape, and what operation literally happened?

Output is [B, S, 4096]. Literally X @ W_q.T — PyTorch transposes the weight (which is stored [out, in]) and matrix-multiplies. Here out = in = 4096, so the shape is unchanged, but in general the first dimension of the weight is the output width.

Why is the MLP "embarrassingly parallel" across tokens but attention is not?

The MLP computes each token's output purely from that same token's input — no token looks at any other. So you can hand different tokens to different GPUs and never need them to talk. Attention is the opposite: a query at position i must consult the keys and values of all tokens up to i. If those tokens live on other GPUs, the GPUs are forced to exchange data. That single difference is why every strategy is interesting only at the attention layer.
