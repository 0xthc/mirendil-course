Sequence parallelism (SP)

The mirror image of TP. Instead of splitting the weights, we split the tokens. Every GPU keeps a full copy of the weights but only handles a chunk of the sequence — which is exactly what kills activation memory.

ELI5

Give each reader (GPU) a different stack of pages from the same book — and every reader keeps a full copy of the dictionary (the weights). Now the activations shrink, because nobody holds the whole book. But there's a twist: the task is "summarize everything up to your page," and to do that a reader needs the earlier pages held by other readers. So they have to pass their pages around (that's the K/V all_gather). The private thinking step (MLP) needs no passing — each reader handles their own pages alone.

The core idea: split the token axis

The input X is [B, S, H]. Under SP, we cut along S: with 8 GPUs and 8192 tokens, GPU 0 gets tokens 0–1023, GPU 1 gets tokens 1024–2047, and so on. Each GPU's local input is [B, S/D, H].

We write X_p for the chunk held by sequence-rank p. Because activations scale with the number of tokens a GPU processes, and each GPU now processes only S/D tokens, activation memory drops by a factor of D. That's the win TP couldn't give.

Terminology cleanup (from the sessions)

Under SP, X_p is neither "row-parallel" nor "column-parallel" — those terms describe how you slice a 2D weight matrix. X_p is an activation sliced along the token axis, so the precise word is sequence-sharded. Use "batch-/sequence-/hidden-/head-sharded" for activations and save row/column-parallel for weights.

Why attention suddenly gets hard

The MLP under SP is trivial — recall it's token-independent. Each GPU runs the full MLP on its own token chunk and is done. No communication at all. (The original session confirmed exactly this: "for SP on MLP, there isn't any all_gather needed right? since it's token-wise" — correct.)

Attention is the problem. Remember causal attention: a query at token i must attend to the keys and values of all tokens 0…i. But under SP those earlier tokens live on other GPUs. GPU 1 holds queries for tokens 1024–2047, but to answer them it needs the keys/values for tokens 0–2047 — and tokens 0–1023 are on GPU 0.

The crux

Local queries need global keys and values. Each GPU has only its local K/V chunk. So before computing attention, every GPU must collect everyone else's K/V chunks. That's an all_gather along the sequence axis.

Queries stay local (each GPU answers its own tokens), but K and V are gathered to full length. Now each GPU can run causal attention for its query rows against the global K/V.
def sp_attn(X_p, W_q, W_k, W_v, W_o, num_heads):
    # X_p is sequence-sharded: [B, S/D, H]. Weights are FULL (replicated).
    q_p = project_to_heads(F.linear(X_p, W_q), num_heads, head_dim)  # [B, heads, S/D, head_dim]
    k_p = project_to_heads(F.linear(X_p, W_k), num_heads, head_dim)
    v_p = project_to_heads(F.linear(X_p, W_v), num_heads, head_dim)

    k = all_gather_seq(k_p)   # [B, heads, S, head_dim]  — full sequence of keys
    v = all_gather_seq(v_p)   # [B, heads, S, head_dim]  — full sequence of values

    attn_p = causal_attn(q_p, k, v)   # local queries vs global K/V -> [B, heads, S/D, head_dim]
    return F.linear(attn_p, W_o)         # [B, S/D, H]  — this rank's output tokens. No all_reduce!

Notice the asymmetry with TP: SP's attention ends with no all_reduce. Each rank owns a distinct set of output tokens, so there's nothing to sum across ranks — every rank's output is final for its own tokens. The communication moved to the middle (the K/V gather) instead of the end.

The causal mask under sharding — a real subtlety

With is_causal=True, SDPA assumes query row i may see key columns 0…i. But on GPU 1, local query row 0 is really global token 1024 — it should see keys 0…1024, not just key 0. So the mask has to account for the rank's offset into the global sequence. The original session flagged exactly this: "this causal masking is correct for tensor-sequence parallelism right?" — and the answer hinges on using rank-relative query positions against global key positions. Get the offset wrong and the model silently attends to the wrong tokens.

Causal load imbalance (and the zigzag fix)

Here's a slick problem that only shows up with causal attention + sequence sharding. If you split the sequence into contiguous chunks, the work is unbalanced:

  • GPU 0 owns the earliest tokens. Token 100 attends to only ~100 keys. Light work.
  • GPU 7 owns the latest tokens. Token 8000 attends to ~8000 keys. Heavy work.

Because attention cost grows with how many earlier tokens you attend to, the GPU with the last chunk does far more work — and everyone waits for the slowest GPU. The fix is zigzag / striped partitioning: instead of giving each GPU a contiguous block, interleave so each GPU gets a mix of early and late tokens. Then every GPU's total attention load is roughly equal.

Analogy

Splitting a book among 8 readers for a "summarize everything up to your page" task. If reader 8 gets the last chapter, they have to recall the whole book; reader 1 just needs page 1. Unfair. Instead, deal pages out like cards (1→A, 2→B, … 9→A, 10→B) so every reader has an even spread of early and late pages. Same total work, evenly shared.

SP's blind spot

SP shrank activations but every GPU still stores all the weights. For a big model that's a lot of wasted replication. TP had the opposite blind spot. Each strategy fixes the other's weakness — which is the whole motivation for combining them.

Check your understanding

Why does the SP MLP need no communication, but SP attention needs an all_gather?

The MLP is token-independent: each token's output depends only on itself, so a GPU with tokens 0–1023 can fully compute their MLP outputs alone. Attention is token-dependent: a query token must attend to all earlier tokens' keys/values, which live on other GPUs — so the K/V chunks must be gathered (all_gather) before attention can run.

TP attention ends with all_reduce; SP attention ends with no collective. Why the difference?

In TP, every rank computes a partial of the same output tokens (row-parallel W_o), so the partials must be summed (all_reduce). In SP, every rank owns a different, disjoint set of output tokens, so each rank's output is already final and complete for its tokens — nothing to combine. SP's communication is in the middle (K/V gather), not at the end.

You split an 8K causal sequence into 8 contiguous 1K chunks and notice GPU 7 is always the bottleneck. Why, and what's the fix?

Causal attention cost grows with position: late tokens attend to far more earlier tokens than early tokens do. GPU 7 holds the latest 1K tokens, so it does the most attention work, and the whole step waits on it. Fix: zigzag/striped partitioning — interleave tokens so each GPU gets a balanced mix of early and late positions, equalizing the load.
