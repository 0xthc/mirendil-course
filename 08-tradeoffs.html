Module 08

Tradeoffs & when to use what

Now the engineering judgment. We'll reason about memory scaling first, then read the test's actual numbers — which tell an honest, non-obvious story.

Memory scaling, as formulas

Let D be your total GPU count. The two memory categories from module 1 scale differently under each strategy:

StrategyModel-state memoryActivation memory
TP1/D ✅ (weights sharded)1 ❌ (full sequence replicated)
SP1 ❌ (weights replicated)1/D ✅ (tokens sharded)
TP + SP1/T1/Σ  (with T·Σ = D)
TSP≈ 1/D ✅≈ 1/D ✅

The TP+SP row is the key comparison. With a fixed budget of D GPUs you must split them between the two axes: D = T · Σ. Spend more on T and you shrink weights more but activations less; spend more on Σ and vice versa. You can't max both — the product is fixed. For D=8 you might pick T=4, Σ=2 (weights ÷4, activations ÷2) or T=2, Σ=4 (weights ÷2, activations ÷4), but never ÷8 on both.

TSP's theoretical pitch

TSP folds both reductions onto the same axis of size D, aiming for both memory classes to scale as ≈1/D simultaneously — the thing TP+SP structurally cannot do with one GPU budget. That's the promise. The catch is communication.

The cost side: communication

Memory savings are never free. Each strategy pays a different communication tax:

  • TP: one all_reduce of a [B, S, H] tensor per attention (and per MLP). Cheap-ish, end of layer.
  • SP: all_gather of K and V every attention. Grows with sequence length.
  • TSP: the most. It broadcasts weight shards during the forward pass and all_gathers K/V — once per loop iteration over D shards. Weight traffic + K/V traffic.

So TSP buys the best memory profile by paying the most data movement. Whether that's a good trade depends entirely on whether the extra traffic rides fast links (NVLink) and can be overlapped with compute. The test's implementation did not overlap it — so its TSP timings are a pessimistic view.

What the test actually measured

From report.md: 8× H100, batch 8, hidden 2K, sweeping sequence length. Here's the peak-memory story for the attention region (lower is better):

Seq lenTP peakSP peakTSP peak
8K1.85 GiB2.84 GiB1.58 GiB
16K3.60 GiB5.59 GiB3.07 GiB
32K7.10 GiB11.09 GiB6.14 GiB
64K14.10 GiB22.09 GiB13.42 GiB

TSP wins on peak memory at every size — exactly the paper's claim. Two more things jump out:

  • SP uses the most memory, not the least. Counterintuitive! The cause is the implementation choice flagged in the report: SP gathers the full K/V sequence onto every rank (all_gather), so each GPU temporarily holds the entire sequence's keys and values — a big buffer. The report names the fix as future work: ring-attention, which streams K/V instead of materializing all of it.
  • The memory advantage of TSP over TP is modest at long sequences (13.42 vs 14.10 GiB at 64K) — because at long context, activations dominate and TSP's edge is mostly on the weight side.

TP is consistently the fastest. It does the least communication and keeps full compute parallelism. TSP and SP are much slower because of their gather/broadcast traffic — and TSP's un-overlapped weight broadcasts make it the slowest at the largest size. This is the honest tradeoff in one picture: TSP buys the lowest memory and pays for it in time.

Read benchmarks like an engineer

These TSP numbers are a floor, not a verdict. The report explicitly lists the optimizations it skipped: async overlap of weight movement, keeping shards resident instead of re-broadcasting, ring-attention for K/V. Each would narrow the runtime gap. The memory win, though, is structural and real regardless. When you evaluate a parallelism strategy, always ask "is this slowness fundamental or just un-optimized?"

The decision guide

Use…When…
TPSequence/activations fit fine and you only need to shrink the weights; you want the fastest option. Great default for moderate context.
SPWeights fit fine but the sequence is huge; activation memory is your wall. (Pair with ring-attention to avoid the K/V-gather blowup this test saw.)
TP + SPYou want independent control over the two reductions and have enough GPUs to spend a 2D grid; or TSP's weight movement can't be hidden on your hardware.
TSPMemory is the bottleneck, context/batch is large, you want both reductions on one GPU budget, and the extra weight traffic stays on fast links or overlaps with compute.
The one-sentence takeaway

TSP is not "strictly better" — it's the right tool when memory is the binding constraint and you can hide the communication. TP wins on speed; SP targets activations specifically; TP+SP gives you a tuning dial; TSP is the memory specialist.

Check your understanding

With D=8 GPUs and TP+SP, can you get both weights and activations down to 1/8? Why is TSP different?

No. TP+SP must split the budget: T·Σ=8. Weights scale as 1/T, activations as 1/Σ, and the product of the divisors is capped at 8 — e.g. ÷4 and ÷2, or ÷2 and ÷4, never ÷8 and ÷8. TSP folds both onto the same axis of size 8, so both target ≈1/8 at once. That simultaneous reduction on one GPU budget is exactly what TSP exists to provide.

The test found SP used the MOST peak memory. Isn't SP supposed to save memory? What happened?

SP does shrink the per-rank input activations, but this implementation's attention all_gathers the full K/V sequence onto every rank — so each GPU briefly materializes the entire sequence's keys and values, a large buffer that dominates peak memory. It's an implementation artifact, not a flaw in the SP idea. The fix named in the report is ring-attention, which streams K/V in chunks instead of gathering all of it at once.

TSP had the lowest memory but among the slowest runtimes. Is that a reason to reject TSP?

Not necessarily. The memory win is structural and real. The slowness is largely un-optimized communication: this implementation re-broadcasts weight shards every forward pass with no overlap, and gathers K/V fully. The report lists concrete fixes (async overlap, resident shards, ring-attention) that would shrink the runtime gap. The engineering question is always "is the cost fundamental or fixable?" — here, much of it is fixable, so TSP stays attractive when memory is the real constraint.
