Module 05

Tensor parallelism (TP)

The first real strategy. Split the weights across GPUs so no GPU holds a full weight matrix. The whole sequence stays replicated everywhere. We shard attention by heads.

ELI5

A thick novel needs proofreading, and attention already splits the job into independent "heads" who never talk to each other. So hand each proofreader (GPU) a different stack of chapters. Nobody needs the whole book; each keeps only their stack. That's tensor parallelism — slice the work and the weights by head. The catch: at the very end you have to staple everyone's corrections back into one book, and that stapling is a quick all-to-all sum.

The core idea: heads are already independent

Recall from module 3 that attention runs num_heads heads in parallel — each head does its own little attention in its own head_dim-wide lane, and the results are concatenated at the end. The heads never talk to each other inside attention. That independence is a gift: we can just give different heads to different GPUs.

With 8 GPUs and 32 heads, each GPU owns 32 / 8 = 4 heads. GPU 0 computes heads 0–3, GPU 1 computes heads 4–7, and so on. Each GPU does 1/8 of the attention work and stores 1/8 of the Q/K/V weights.

Column-parallel vs row-parallel: the only hard part

Here's the question that generated the most back-and-forth in the original sessions: Q/K/V and the output projection W_o are all square [H, H] matrices — why can't I shard them the same way? The answer is that they sit on opposite sides of the attention operation, so they must be sliced along different axes.

Reason from input/output axes (not from the visual "rows look like columns" trap). Remember F.linear(X, W) computes X @ W.T with W stored as [out_features, in_features].

Column-parallel — split the OUTPUT features (used for Q, K, V)

The Q/K/V projections come before attention. Their job is to produce the per-head Q/K/V. Since GPU 0 only needs the Q/K/V for its heads, it only needs the slice of W_q that produces those output features. We split the output dimension:

W_q is [H, H] = [out, in].  Slice the OUT (head) dimension:
   W_q_p on rank p = W_q[p*chunk : (p+1)*chunk, :]   # shape [H/D, H]
   X @ W_q_p.T  ->  [B, S, H/D]    # only this rank's heads

"Column-parallel" because, in the math convention where you'd write X·W, you're splitting W's output columns. Each rank produces a different, complete slice of the output — no summing needed yet. Input X is replicated, so every rank can compute its slice independently.

Row-parallel — split the INPUT features (used for W_o)

The output projection W_o comes after attention. Its input is the concatenated head outputs. But GPU 0 only has its own heads' outputs — a partial input. So GPU 0 takes the slice of W_o that consumes those input features. We split the input dimension:

W_o is [H, H] = [out, in].  Slice the IN (head) dimension:
   W_o_p on rank p = W_o[:, p*chunk : (p+1)*chunk]   # shape [H, H/D]
   attn_p @ W_o_p.T  ->  [B, S, H]   # a PARTIAL full-width output

Now each rank produces a [B, S, H] tensor that is the partial contribution of its heads to the final answer. The true output is the sum of all ranks' partials. That's the all_reduce.

The pattern that recurs forever

Column-parallel (split output) → no communication, each rank holds a distinct slice. Row-parallel (split input) → produces partial sums → needs an all_reduce to combine. Attention is column-parallel Q/K/V followed by row-parallel W_o. The all_reduce at the end is the price of having split W_o by input.

The insight everyone trips on: head-space vs residual-space

This was singled out in the teaching notes as "the user's main gap," so it's worth slowing down. Before W_o, the attention output is in head space — it's per-head features, not yet a meaningful update to the token's residual vector. W_o is the thing that translates head-space features into residual-space updates. So when you shard heads, you're sharding head-space; each rank's head-space slice produces a partial residual-space vector, and those partials sum.

Concretely, view it as a block sum. With 4 heads, write the attention output as four side-by-side blocks and W_o as four stacked blocks:

attn = [A0 | A1 | A2 | A3]      each Ai: [B, S, head_dim]
W_o  = [W0 ; W1 ; W2 ; W3]      each Wi: [head_dim, H]

attn @ W_o = A0@W0 + A1@W1 + A2@W2 + A3@W3      # a SUM of per-head terms!

Each term Ai @ Wi is a full [B, S, H] residual update from one head. The output projection is secretly a sum over heads. So if rank 0 owns heads 0–1 it computes A0@W0 + A1@W1, rank 1 owns heads 2–3 and computes A2@W2 + A3@W3, and all_reduce adds the two partials. The math of TP is just rearranging this sum across GPUs.

The whole thing in code

Here's tp_attn from tensor_parallelism.py. Every line should now read as obvious:

def tp_attn(X, W_q_p, W_k_p, W_v_p, W_o_p, p_heads):
    B, S, H = X.shape           # X is REPLICATED: full [B, S, H] on every rank

    # column-parallel Q/K/V: each produces only this rank's heads
    q = project_to_heads(F.linear(X, W_q_p), p_heads, head_dim)   # [B, p_heads, S, head_dim]
    k = project_to_heads(F.linear(X, W_k_p), p_heads, head_dim)
    v = project_to_heads(F.linear(X, W_v_p), p_heads, head_dim)

    attn_p = F.scaled_dot_product_attention(q, k, v, is_causal=True)
    attn_p = attn_p.transpose(1, 2).contiguous().view(B, S, p_hidden)

    out = F.linear(attn_p, W_o_p)        # row-parallel: PARTIAL [B, S, H]
    dist.all_reduce(out, op=dist.ReduceOp.SUM)   # sum partials -> true output
    return out

TP for the MLP (same pattern, even simpler)

The MLP is W_in (expand H→I) then W_out (contract I→H). Same recipe: column-parallel the first matrix (split the intermediate dimension I — each rank computes a slice of the expanded hidden), row-parallel the second (each rank consumes its slice of I and produces a partial output), then all_reduce. Expand-then-contract maps perfectly onto column-then-row.

Play with the numbers. Notice for TP that X stays [B, S, H] on every rank (activations are not reduced — that's TP's whole limitation), while the weights and the per-head tensors shrink by D.

What TP did and didn't buy you

Look at the TP rows in the calculator. The weights and per-head Q/K/V shrank by D — model-state memory win. But the replicated X and the full-sequence attention are unchanged — zero activation-memory win. That gap is precisely what sequence parallelism (next module) exists to close.

Check your understanding

Q/K/V and W_o are all [H, H]. Why can't you reuse the same sharding function for both?

Because the slice axis differs. Q/K/V are column-parallel (split the output/head dimension — the rows of the stored [out, in] weight), since their input X is replicated and each rank should produce a distinct slice of output. W_o is row-parallel (split the input dimension — the columns of the stored weight), since its input is already a per-head partial and each rank should consume only its heads. Same shape, opposite axis — so you need two different slicing helpers (shard_col_parallel vs shard_row_parallel in the code).

Why does TP attention end with all_reduce rather than all_gather?

Because the row-parallel W_o makes each rank produce a partial sum of the same full-width [B,S,H] output — the per-head contributions that must be added. all_gather would concatenate distinct pieces, which is wrong here: the pieces overlap and represent the same positions. You want all_reduce(SUM).

You scale to S = 64K with TP on 8 GPUs and still run out of memory. Did TP fail? What's the fix?

TP didn't fail at its job — it shrank the weights — but it never touches activation memory, and at 64K tokens the activations (full sequence replicated on every GPU) are what's blowing up. The fix is to also shard the sequence: that's Sequence Parallelism (module 6), and ultimately TSP (module 7), which shrinks both at once.
