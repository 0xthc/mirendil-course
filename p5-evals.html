Evals & telemetry

An LLM feature can feel great in a demo and still be quietly broken in production. This page is about the discipline that tells you the truth: how to know whether a feature is good, and whether a change actually helped — before you ship it and after.

ELI5

Imagine you tweak a cookie recipe and ask three friends — they all say it tastes better, so you bake 10,000 for a party. That's how most people "test" a prompt change. The professional way is to keep a fixed tasting panel of 50 cookies you score the same way every time, write down the scores for the old recipe and the new one, and only switch if the new scores are higher. Evals are that fixed tasting panel for LLM features. Telemetry is watching what real party guests actually eat.

Why you need evals at all

The same prompt, sent twice, can return different text. Temperature, model updates, a reworded instruction, a new retrieval source — any of these can shift behavior in ways you won't notice by eyeballing a handful of outputs. So the core problem is:

  • Outputs are non-deterministic. "It worked when I tried it" is one sample from a distribution. You need many samples to see the distribution.
  • "Looks good on my 3 examples" is not evidence. Three examples can't catch a regression that hits 8% of inputs. That 8% is still thousands of unhappy users.
  • Every change needs a before/after signal. Prompt edits and model upgrades both look like improvements in your head. Only a measured comparison tells you which way the needle actually moved — and whether you broke something else while fixing the thing you cared about.
The whole idea in one line

An eval turns "I think this is better" into a number you can compare. Without that number, you are shipping vibes. With it, every prompt and model change becomes a measurable experiment with a pass/fail result.

Offline eval sets

An offline eval is a curated dataset you run before shipping — no live users involved. Each item is an input, usually paired with either an expected output or a rubric describing what a good answer looks like. A few flavors you should be able to name:

  • Golden set — a small, hand-blessed set of inputs with known-good answers. The cases you would be embarrassed to get wrong. Quality over quantity.
  • Regression set — every bug you've ever fixed, frozen as a test case. If a future change reintroduces an old failure, this catches it. It only grows.
  • Coverage set — a broader sample spanning the real input distribution: edge cases, languages, long inputs, adversarial phrasings, the boring 80% too.

The high-leverage move is to wire the regression set into CI: a prompt or model change cannot merge unless the eval score clears a threshold. This is the same instinct as unit tests gating a deploy — except the "assertion" is a score over a dataset rather than a single boolean. It's how a serious product team keeps quality from silently drifting down over months of well-intentioned tweaks.

Analogy

A golden set is your smoke test — small, fast, must always pass. A regression set is your bug-fix test suite — it never forgets a mistake. A coverage set is your integration suite — slower, broader, catches the surprises. Same three layers you already trust in normal software, pointed at a non-deterministic model.

How do you score an output?

Building the dataset is half the job; the other half is deciding what counts as correct. Scoring methods sit on a ladder from cheap-and-rigid to expensive-and-flexible:

MethodWhat it checksGood forWatch out for
Exact / structuralString match, regex, valid JSON, schema conforms, code compilesClassification, extraction, structured outputs, tool-call argsToo strict for open-ended text — a correct paraphrase fails
Rubric scoringPoints against explicit criteria (cites a source? answers the question? stays on policy?)Multi-dimensional quality on free textNeeds a clear rubric or scores get noisy
LLM-as-judgeA strong model grades the output (score or A/B preference) against instructionsOpen-ended quality at scale, where humans are too slowThe judge has biases and needs its own validation (see below)
Human reviewA person reads and ratesThe gold standard; calibrating the other methodsSlow and expensive — use to validate cheaper graders, not for every run

The practical pattern: use the cheapest method that captures what you care about. Structural checks for anything machine-verifiable (always prefer these — they're free and unambiguous), rubric or LLM-judge for the squishy stuff, and a thin layer of human review to keep the automated graders honest.

LLM-as-judge, and why to distrust it

Using a capable model (say Opus 4.8) to grade another model's output is powerful: it scales open-ended judgment to thousands of cases for the price of a few API calls. But a judge is itself an LLM feature, so it has failure modes you must actively defend against:

  • Position bias — in A/B comparisons, judges often favor whichever answer is shown first. Fix: run both orderings and average, or randomize.
  • Verbosity / style bias — judges tend to reward longer, more confident, more nicely formatted answers even when they're not more correct.
  • Self-preference — a judge may prefer outputs from its own model family.
  • It needs its own eval. Before trusting a judge, measure its agreement with human labels on a sample. If the judge agrees with humans only 60% of the time, its scores are barely better than noise.
Gotcha

The most common self-inflicted wound: trusting an LLM judge you never validated. Teams wire up a judge, watch the score go up, and ship — without ever checking that the judge's grades correlate with what humans (or users) actually consider better. An unvalidated judge can confidently rank a regression as an improvement. Always calibrate the judge against human labels first, and re-check it when you change the judge model.

Telemetry: measuring in production

Offline evals tell you about a fixed dataset. Real users do things your dataset never imagined, so you also instrument the live product. The signals worth collecting, roughly from weak to strong:

  • Explicit feedback — thumbs up/down, star ratings. Easy to collect but sparse and biased (angry users vote more than happy ones).
  • Implicit behavior — acceptance rate (did they keep the suggestion?), edit rate (how much did they change it?), retry/regenerate rate, copy actions. Far more honest than stars because it's what people did, not what they said.
  • Task success — did the user actually accomplish their goal? The thing you ultimately care about, and the hardest to measure. Often proxied (did the support ticket close? did the code get committed?).
  • Operational — latency (p50/p95), error rate, cost per request. A feature that's "smarter" but twice as slow can still lose users.
Vanity metrics mislead

A metric is vain when it goes up while users get worse. Classics: thumbs-up rate climbs because you only show the button to users who already finished happily; total messages per session rises because the assistant got confusing and people have to re-ask; acceptance rate looks great because the bar for "accepted" is one click and people click to dismiss. Always ask: if this number doubled, would a user actually be better off? Tie metrics to real outcomes (task completed, ticket resolved, time saved), not activity.

Building the loop

Offline evals and production telemetry aren't two separate things — they feed each other. Production surfaces failures you didn't anticipate; those failures become new eval cases; the eval set gates the next change; telemetry confirms the change helped real users. That cycle is the engine of a feature that improves over time instead of decaying.

offline evals → gate the deploy → ship → telemetry catches real-world failures → mine failures into new eval cases → back to offline evals

The compounding loop: every production surprise becomes a permanent regression test. The eval set is a living record of everything your feature must never get wrong again.

This is the same philosophy the rest of this track keeps returning to: validate small, measure real. Prove it on a tight, trusted dataset before you ship; then watch what actually happens with real users, and fold what you learn back into the dataset.

How this shows up in the interview

Product-engineering interviews probe whether you can tell signal from vibes. Expect questions like these.

How do you know a prompt change improved the product?

Not by looking at a few outputs — those are single samples from a non-deterministic distribution. I'd run both the old and new prompt over a fixed eval set and compare a full scorecard (accuracy/structural checks plus rubric or a validated LLM-judge for open-ended quality), looking for regressions on metrics I wasn't targeting, not just the one I was. If the candidate clears every threshold, I ship it behind a flag and confirm with production telemetry — acceptance/edit rate and task success — that real users are better off, not just the offline number. "Improved" means: the scorecard didn't regress and a real-world outcome moved.

Design an eval harness for an LLM feature.

Five pieces. (1) Dataset: a golden set of must-pass cases, a regression set seeded from every past bug, and a coverage set sampling the real input distribution including edge cases. (2) Runner: send every input through the candidate version, capture raw outputs, control temperature for reproducibility. (3) Scorers: structural checks for anything machine-verifiable (valid JSON, schema, exact labels), rubric or LLM-judge for open-ended quality — and validate the judge against human labels first. (4) Aggregation: roll per-item scores into per-metric numbers, with confidence intervals so I don't over-read noise. (5) Gate: wire it into CI so a change can't merge unless every metric clears its threshold. Then close the loop by mining production failures into new dataset cases.

Which metric tells you it's working — and why do vanity metrics lie?

The metric closest to the user's actual goal: task success (ticket resolved, code committed, answer accepted-and-kept), backed by implicit behavior like edit rate. Vanity metrics lie because they measure activity, not outcomes, and often move in the wrong direction relative to value: thumbs-up rate can be inflated by who you ask; messages-per-session can rise precisely because the assistant got confusing and users re-ask; acceptance rate can look great when "accept" is one click people use to dismiss. The test for any metric: if it doubled, would a user genuinely be better off? If not, it's vanity. Anchor on outcomes, and triangulate several signals rather than trusting one.

Check your understanding

Your new prompt scores higher on helpfulness but the eval also shows format-valid dropping from 98% to 82%. Ship it?

No — that's a regression. The 16-point drop in format-valid means roughly one in six outputs no longer parses, which will break whatever downstream code consumes the structured output. A win on one metric never justifies an unexplained drop on another. Either fix the prompt so it keeps the format and the helpfulness gain, or reject it. This is exactly why you score the whole scorecard, not just the metric you set out to improve.

Why isn't an LLM-as-judge score trustworthy on its own?

Because the judge is itself an LLM with its own biases — position bias (favoring the first option shown), verbosity/style bias (rewarding longer, prettier answers), and possible self-preference for its own model family. A judge you haven't validated can rank a regression as an improvement with total confidence. Before trusting it, measure its agreement with human labels on a sample; mitigate position bias by averaging both orderings; and re-validate whenever you change the judge model. The judge is a tool that needs its own eval.

Thumbs-up rate on your feature is climbing month over month. Is the feature getting better?

Not necessarily — that's a candidate vanity metric. The rate can climb for reasons unrelated to quality: maybe the button only appears after a successful flow (survivorship), maybe a UI change made the thumbs-up easier to hit, maybe your happiest cohort grew. Confirm it against outcome-based signals — task success, edit rate, retention — and check whether the offline eval scorecard agrees. If the activity metric rises while outcome metrics stay flat or fall, users aren't better off.
