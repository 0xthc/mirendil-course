Module 04

Multi-GPU primitives

As a full-stack engineer, this module is the one that'll feel most like home: it's distributed-systems programming. A handful of processes, each owning one GPU, coordinating through a small set of collective operations.

The mental model: one process per GPU

You do not write one program that drives 8 GPUs. You write one program that runs 8 times — once per GPU — and the copies talk to each other. This is the Single-Program-Multiple-Data (SPMD) model. If you've written code with MPI, or even just thought about how 8 identical web server replicas coordinate through a shared channel, you already have the right instinct.

# You launch it like this — torchrun spawns 8 copies of the script:
torchrun --standalone --nproc_per_node=8 tensor_parallelism.py

Each copy is a rank. Three environment variables tell a copy who it is:

VariableMeaning
RANKThis process's global ID, 0 … world_size-1. Rank 0 is conventionally the "leader".
LOCAL_RANKWhich GPU on this physical machine this process should use. On a single 8-GPU box, LOCAL_RANK == RANK.
WORLD_SIZETotal number of processes (= total GPUs). Here, 8. This is the D from module 1.

Here's the real setup code from the repo. Read it as "figure out who I am, bind myself to my GPU, and join the group":

def setup_distributed():
    dist.init_process_group(backend="nccl")   # join the group; NCCL = NVIDIA's GPU comms library

    rank = dist.get_rank()                  # who am I globally?
    world_size = dist.get_world_size()      # how many of us are there?
    LOCAL_RANK = int(os.environ["LOCAL_RANK"])

    torch.cuda.set_device(LOCAL_RANK)       # claim my GPU
    device = torch.device("cuda", LOCAL_RANK)
    return rank, world_size, LOCAL_RANK, device
NCCL

backend="nccl" selects NVIDIA's Collective Communications Library — the thing that actually moves tensors GPU-to-GPU over fast links (NVLink). You never call NCCL directly; you call the dist.* collectives below and NCCL does the wire-level work. (On CPU-only test runs the repo uses the gloo backend instead.)

The three collectives — your entire toolbox

Astonishingly, the whole course is built from just three communication operations. A collective is an operation that every rank calls together. Each one answers a different question: "how do I share data across ranks?"

broadcast — one-to-all

One rank has a tensor; everyone else gets an identical copy.

rank0
[ABCD]
rank1
[ ? ]
rank2
[ ? ]
rank3
[ ? ]
↓ broadcast(src=0) ↓
rank0
[ABCD]
rank1
[ABCD]
rank2
[ABCD]
rank3
[ABCD]
Used at setup to give every GPU the same starting weights, and in TSP to circulate weight shards during the forward pass.
dist.broadcast(tensor, src=0)   # rank 0 fills tensor; all others receive into it

Subtle but important: every rank must call broadcast. The source rank supplies the data; the others pass an empty buffer of the right shape to receive into. If even one rank skips the call, the whole group deadlocks (more on that below).

all_reduce — combine-and-share

Every rank has a tensor of the same shape but different values. all_reduce combines them element-wise (usually by summing) and gives everyone the combined result.

rank0
[1,0,0]
rank1
[0,2,0]
rank2
[0,0,3]
rank3
[1,1,1]
↓ all_reduce(SUM) ↓
rank0
[2,3,4]
rank1
[2,3,4]
rank2
[2,3,4]
rank3
[2,3,4]
Each rank computed a partial result; summing the partials gives the true full result. This is exactly the TP attention output step.
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)   # in-place: tensor becomes the sum across ranks

Use this when each rank has computed part of a sum and the true answer is the total. The shapes match; you're adding numbers in corresponding positions.

all_gather — collect-the-pieces

Every rank has a different chunk of some larger tensor. all_gather concatenates all the chunks and gives everyone the full assembled tensor.

rank0
[A]
rank1
[B]
rank2
[C]
rank3
[D]
↓ all_gather ↓
rank0
[ABCD]
rank1
[ABCD]
rank2
[ABCD]
rank3
[ABCD]
Pieces are concatenated, not summed. This is exactly how SP reassembles the full K/V sequence from per-GPU token chunks.
gathered = [torch.empty_like(chunk) for _ in range(world_size)]
dist.all_gather(gathered, chunk)        # gathered[i] = the chunk that rank i contributed
full = torch.cat(gathered, dim=...)     # then concatenate along the right axis
The distinction that matters most

all_reduce sums overlapping values; all_gather concatenates distinct pieces. Choosing wrong is the classic bug. If each rank computed a partial sum of the same thing → all_reduce. If each rank holds a different slice that needs to sit side by side → all_gather. TP's output uses all_reduce; SP's K/V uses all_gather. Keep that pair in mind.

The deadlock rule (the one bug you'll actually hit)

Collectives are synchronization points. Every rank must call the same collectives, in the same order. If rank 0 calls all_gather while rank 1 calls all_reduce — or rank 1 skips a call inside an if branch — the group hangs forever, silently. There's no crash, just a stuck program.

This actually happened in the sessions

The original SP implementation "was hanging." The cause was exactly this class of bug — a collective-call mismatch across ranks. The fix is discipline: issue collectives unconditionally and identically on every rank. Later, in TSP, you'll see a loop for r in range(world_size) that broadcasts weights each iteration — and the explicit constraint that all ranks must run that loop the same number of times in the same order, or it deadlocks.

Why broadcast weights at setup?

A pattern you'll see everywhere: rank 0 creates the full random weights and input, then broadcasts them so every rank starts identical. Then each rank slices out its own shard locally. Why not just let each rank generate its own shard with the same random seed?

The answer (a real question from the sessions)

If every rank used the same seed, every rank would generate the same shard — defeating the point of sharding (you'd have 8 copies of slice 0 and no copies of the others). And if they used different seeds, the shards wouldn't fit together into one coherent weight matrix, so you couldn't validate against a single-GPU reference. The clean teaching approach: one rank makes the whole true weight, broadcasts it, and each rank deterministically slices out a different piece. (In real production you'd load only your shard from disk and skip the broadcast — that's listed as future work in the report.)

Check your understanding

Each of 4 GPUs computed a [B,S,H] tensor that represents one-quarter of a sum. You want the full result on every GPU. Which collective?

all_reduce(SUM). The four tensors have the same shape and represent partial sums of the same quantity; summing them element-wise gives the true total, and all_reduce leaves that total on every rank.

Each of 4 GPUs holds the keys for a different 2K-token chunk of an 8K sequence. You want the full 8K keys on every GPU. Which collective?

all_gather. The chunks are distinct slices that must sit side by side to reconstruct the 8K sequence — concatenation, not summation. (Then a torch.cat, or better, gather straight into a preallocated buffer — an optimization the repo actually made.)

Your 8-rank program runs fine on 7 ranks but rank 3 took an early return inside an if. What happens, and why is it hard to debug?

The program hangs forever. The other 7 ranks reach a collective and wait for rank 3 to join it, but rank 3 already left. No exception is raised — it just deadlocks silently. That's why the rule is "every rank issues every collective, identically, in the same order," and why conditional logic around collectives is dangerous.
