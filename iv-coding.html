Interview prep

Coding exercises

Six realistic, LLM-flavored coding rounds for a product-engineering loop. Each one is the kind of small-but-real system you actually ship around a model: streaming, agent loops, validation, rate limits, evals, and caching. Try each before opening the solution — the interviewer is watching how you think, not just whether it compiles.

ELI5

An LLM coding round is less like a puzzle and more like a tiny on-the-job task: “here is a model, here is a half-real problem, wire it up like you would in production.” The model is a black box that streams text, sometimes returns junk, and sometimes errors. Your job is to be the calm, careful plumbing around it — handle the happy path in five minutes, then spend the rest of the time on what happens when things go wrong.

How coding rounds run at an AI lab

Product-engineering rounds at AI labs are deliberately practical. You get a real-ish task, a shared editor or your own machine, and 45–60 minutes. The model call is usually stubbed or behind a thin client so you are not graded on prompt-wrangling — you are graded on the engineering around the model: streaming, cancellation, retries, validation, backpressure, caching, observability.

Three habits separate strong candidates:

  • Talk as you go. Narrate assumptions and trade-offs. Silence reads as “stuck,” even when you are thinking clearly.
  • Get the happy path working fast, then attack failure. A working skeleton in ten minutes buys you the credibility to spend thirty on edge cases — which is where the points are.
  • Name the production concerns out loud even if you do not implement all of them: timeouts, cancellation, idempotency, metrics, cost.
What they are actually grading

They are grading judgment and failure handling, not “did it pass the one test.” Anyone can call a model and print the result. The signal is: do you cancel cleanly on disconnect, cap your retries, validate untrusted output, and degrade gracefully when the model misbehaves? Treat the model as an unreliable network dependency that occasionally lies, and you will out-perform people writing cleverer algorithms.

The #1 way people lose the round

Ignoring the failure paths. The model returns malformed JSON and your parser throws and crashes the request. The client disconnects and your server keeps generating (and billing) tokens into the void. Your retry loop has no cap and hammers a rate-limited API forever. Write the happy path, then immediately ask yourself “what breaks?” — out loud.

Below, each exercise shows the problem, clarifying questions, hints, and a full worked solution behind a toggle. Open the toggle only after you have sketched your own approach.

The shape of every solution

Most of these follow the same arc: a request comes in, you do some work against the model, and you have to handle cancellation, errors, or retries along the way. Here is that lifecycle as a stepper — keep it in your head as you read each problem.

Exercise 1 — Streaming chat endpoint with cancellation

Problem. Build an HTTP endpoint that streams a model’s tokens to the client as they arrive (Server-Sent Events). If the client disconnects or aborts mid-stream, you must stop generating immediately and clean up — no orphaned model calls, no leaked resources.

Clarifying questions / constraints

  • Transport — SSE, chunked HTTP, or WebSocket? (Assume SSE; it is the common default for chat.)
  • How does the model client expose streaming — async iterator with an abort signal? (Assume yes; that is the modern shape.)
  • Do we need to persist the partial completion on disconnect, or just discard? (Assume discard for now, but mention persistence.)
  • What should the client see on error mid-stream — an SSE error event then close? (Yes.)

Hints

  • Wire the request’s abort/close signal straight into the model call’s AbortController.
  • Use a try/finally so cleanup runs whether the stream finished, errored, or was cancelled.
  • Send a final sentinel event ([DONE]) so the client knows the stream ended cleanly vs. dropped.
Show full solution + what interviewers look for

TypeScript / Node (works with the Web Response streaming API used by modern runtimes). The model call is a clearly-marked fake that yields tokens.

// --- FAKE model client (replace with real SDK, e.g. Anthropic streaming) ---
// Real usage: client.messages.stream({ model: "claude-sonnet-4-6", ... })
async function* fakeModelStream(
  prompt: string,
  signal: AbortSignal,
): AsyncGenerator<string> {
  const tokens = ("Thinking about: " + prompt).split(" ");
  for (const t of tokens) {
    if (signal.aborted) throw new DOMException("aborted", "AbortError");
    await new Promise((r) => setTimeout(r, 40)); // simulate token latency
    yield t + " ";
  }
}

// --- The endpoint: returns a streaming Response ---
export async function handleChat(req: Request): Promise<Response> {
  const { prompt } = await req.json();
  if (typeof prompt !== "string" || !prompt.trim()) {
    return new Response("prompt required", { status: 400 });
  }

  // Bridge the client disconnect -> model abort.
  const ac = new AbortController();
  req.signal.addEventListener("abort", () => ac.abort());

  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    async start(controller) {
      const send = (event: string, data: string) =>
        controller.enqueue(encoder.encode(`event: ${event}\ndata: ${data}\n\n`));
      try {
        for await (const token of fakeModelStream(prompt, ac.signal)) {
          send("token", JSON.stringify({ text: token }));
        }
        send("done", "[DONE]");
      } catch (err) {
        if ((err as Error)?.name === "AbortError") {
          // Expected on disconnect — do NOT treat as a server error.
          return;
        }
        send("error", JSON.stringify({ message: "generation failed" }));
      } finally {
        ac.abort(); // stop the model if we exited any other way
        try { controller.close(); } catch {}
      }
    },
    cancel() {
      ac.abort(); // reader went away (client closed the tab)
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache, no-transform",
      "Connection": "keep-alive",
    },
  });
}

What interviewers look for.

  • Cancellation actually propagates. The single most important line is bridging req.signal into the model’s AbortController, plus the cancel() hook on the stream. Without it the model keeps generating (and billing) after the client leaves.
  • AbortError is not an error. Distinguishing an expected abort from a real failure shows production experience — logging aborts as 500s is a classic false alarm.
  • finally for cleanup. Resources released on every exit path: completion, error, or cancel.
  • Protocol clarity. A done sentinel and a distinct error event let the client tell “finished” from “dropped.” Bonus points for mentioning heartbeats to keep proxies from killing idle connections, and persisting the partial on disconnect if the product needs resumability.

Exercise 2 — Tool-calling agent loop

Problem. The model can request tool calls. Implement the loop: send the conversation, if the model returns tool calls, dispatch each to a registry of tools, feed the results back, and repeat until the model returns a final text answer. Guard against infinite loops with a max-steps cap, and handle a tool that throws.

Clarifying questions / constraints

  • Can the model request multiple tools in one turn? (Assume yes — dispatch them, ideally in parallel.)
  • On tool error, do we abort or feed the error back to the model? (Feed it back as a tool result so the model can recover or apologize.)
  • What is the max-steps behavior — error, or return best-effort? (Stop and return a graceful message.)
  • Are tools trusted? Validate tool arguments before executing? (Yes — model-chosen args are untrusted input.)

Hints

  • Model the loop as: append assistant turn → if tool calls, run them → append tool results → loop.
  • A tool registry is just Map<name, (args) => Promise<result>> plus arg validation.
  • Wrap each tool call so a throw becomes a structured error result, not a crash.
Show full solution + what interviewers look for

Python, async. The model client is a fake that scripts a tool call then a final answer.

import asyncio, json
from dataclasses import dataclass
from typing import Any, Awaitable, Callable

# --- Tool registry -------------------------------------------------
@dataclass
class Tool:
    name: str
    run: Callable[[dict], Awaitable[Any]]
    validate: Callable[[dict], None]  # raises ValueError on bad args

async def get_weather(args: dict) -> dict:
    return {"city": args["city"], "temp_c": 21}

def _validate_weather(args: dict) -> None:
    if not isinstance(args.get("city"), str) or not args["city"]:
        raise ValueError("city must be a non-empty string")

REGISTRY = {"get_weather": Tool("get_weather", get_weather, _validate_weather)}

# --- FAKE model: turn 1 asks for a tool, turn 2 answers -----------
# Real usage: await client.messages.create(model="claude-opus-4-8", tools=[...])
async def fake_model(messages: list) -> dict:
    if not any(m["role"] == "tool" for m in messages):
        return {"type": "tool_calls",
                "calls": [{"id": "c1", "name": "get_weather",
                           "args": {"city": "Paris"}}]}
    return {"type": "text", "text": "It is 21C in Paris."}

# --- The agent loop ----------------------------------------------
async def dispatch(call: dict) -> dict:
    tool = REGISTRY.get(call["name"])
    if tool is None:
        return {"tool_call_id": call["id"], "error": "unknown tool"}
    try:
        tool.validate(call["args"])
        result = await tool.run(call["args"])
        return {"tool_call_id": call["id"], "result": result}
    except Exception as e:  # feed the error back, do not crash the loop
        return {"tool_call_id": call["id"], "error": str(e)}

async def run_agent(user_msg: str, max_steps: int = 6) -> str:
    messages = [{"role": "user", "content": user_msg}]
    for step in range(max_steps):
        reply = await fake_model(messages)
        if reply["type"] == "text":
            return reply["text"]
        # record the assistant's tool request, then run the tools
        messages.append({"role": "assistant", "tool_calls": reply["calls"]})
        results = await asyncio.gather(*(dispatch(c) for c in reply["calls"]))
        for r in results:
            messages.append({"role": "tool", "content": json.dumps(r)})
    return "Stopped: hit max steps without a final answer."  # graceful cap

What interviewers look for.

  • The loop terminates. A max_steps cap is non-negotiable — a model can loop forever requesting tools. Returning a graceful message beats raising.
  • Tool errors feed back, not up. Catching the exception and returning it as a tool result lets the model recover. Crashing the whole request on one bad tool call is the junior move.
  • Untrusted arguments. Validating args before executing — the model picked those values, treat them like user input. Unknown tool names handled too.
  • Parallel dispatch. asyncio.gather for multiple calls in one turn shows you think about latency. Mention idempotency and per-tool timeouts as next steps.

Exercise 3 — Structured output: parse, validate, retry

Problem. You ask the model for JSON matching a schema. The response may be malformed JSON, or valid JSON that fails the schema. Parse it; on failure, send a corrective message back to the model and retry, up to N times. If it still fails, give up gracefully with a clear error.

Clarifying questions / constraints

  • What is the schema and how strict? (Assume a small typed schema; reject extra/missing fields.)
  • Does the model wrap JSON in prose or code fences? (Often yes — extract before parsing.)
  • How many retries, and do we change temperature on retry? (N=3; lower temperature and quote the exact error.)
  • What does “give up gracefully” mean — exception or a typed failure result? (Typed result the caller can branch on.)

Hints

  • Separate the two failure modes: parse error vs. validation error. The corrective message differs.
  • Feed the actual error text back to the model — specific feedback fixes things faster than “try again.”
  • Many providers support a forced-JSON / tool-schema mode; mention it, but still validate.
Show full solution + what interviewers look for

Python with a small hand-rolled validator (so it is self-contained; in production use pydantic). The model fake returns junk first, then valid JSON.

import json, re
from dataclasses import dataclass

@dataclass
class Ticket:
    title: str
    priority: str   # one of: low, medium, high

def validate_ticket(obj: dict) -> Ticket:
    if not isinstance(obj.get("title"), str) or not obj["title"].strip():
        raise ValueError("field 'title' must be a non-empty string")
    if obj.get("priority") not in {"low", "medium", "high"}:
        raise ValueError("field 'priority' must be one of low|medium|high")
    return Ticket(obj["title"], obj["priority"])

def extract_json(text: str) -> dict:
    # strip code fences / surrounding prose, then parse
    fenced = re.search(r"\{.*\}", text, re.DOTALL)
    if not fenced:
        raise ValueError("no JSON object found in response")
    return json.loads(fenced.group(0))  # may raise JSONDecodeError

# --- FAKE model: bad output first, then good ----------------------
# Real usage: client.messages.create(model="claude-sonnet-4-6", ...)
_SCRIPT = [
    "Sure! {title: 'Login broken', priority: 'urgent'}",  # not valid JSON
    '{\"title\": \"Login broken\", \"priority\": \"urgent\"}',    # bad enum
    '{\"title\": \"Login broken\", \"priority\": \"high\"}',       # valid
]
def fake_model(messages: list) -> str:
    return _SCRIPT[min(len([m for m in messages if m["role"] == "user"]) - 1, 2)]

def get_ticket(prompt: str, max_retries: int = 3):
    messages = [{"role": "user", "content": prompt}]
    last_err = None
    for attempt in range(max_retries):
        raw = fake_model(messages)
        try:
            return validate_ticket(extract_json(raw))  # success
        except (ValueError, json.JSONDecodeError) as e:
            last_err = str(e)
            # corrective turn: quote the EXACT error so the model can fix it
            messages.append({"role": "assistant", "content": raw})
            messages.append({"role": "user", "content":
                f"That response was invalid: {last_err}. "
                "Reply with ONLY valid JSON: "
                '{\"title\": string, \"priority\": \"low\"|\"medium\"|\"high\"}'})
    return {"ok": False, "error": f"gave up after {max_retries} tries: {last_err}"}

What interviewers look for.

  • Two distinct failure modes. Parse failure and schema failure are different; both are caught and both produce a useful corrective message.
  • Specific corrective feedback. Echoing the exact error text (“priority must be one of…”) is what makes the retry converge. Vague “try again” wastes attempts.
  • Bounded retries + graceful give-up. Returns a typed failure the caller can handle instead of throwing into the void or looping forever.
  • Robust extraction. Handling prose/fences around the JSON shows real-world experience. Bonus: mention native structured-output / tool-schema modes and validating even then, plus lowering temperature on retry.

Exercise 4 — Token-bucket rate limiter

Problem. Implement a per-API-key rate limiter using the token-bucket algorithm. Each key gets a bucket that refills over time. allow(key) returns whether the request proceeds and, if not, a retry_after in seconds. It must be safe under concurrency.

Clarifying questions / constraints

  • Token bucket (allows bursts) or fixed window? (Token bucket — smooth + burst-tolerant, the standard for APIs.)
  • Single process or distributed across servers? (Assume single process; mention Redis for distributed.)
  • Per-key rate and burst capacity configurable? (Yes — rate and capacity as params.)
  • Lazy refill on access, or a background timer? (Lazy — compute elapsed time on each call; no timer to leak.)

Hints

  • Store per key: current token count and last-refill timestamp. Refill = elapsed × rate, capped at capacity.
  • Refill lazily on each allow call — no background thread needed.
  • retry_after = how long until one token is available = (1 - tokens) / rate.
Show full solution + what interviewers look for

Python with a lock for thread-safety. The same lazy-refill math works in async by swapping the lock for an asyncio.Lock.

import time, threading
from dataclasses import dataclass

@dataclass
class Decision:
    allowed: bool
    retry_after: float = 0.0

class TokenBucketLimiter:
    """Per-key token bucket. rate = tokens/sec, capacity = max burst."""
    def __init__(self, rate: float, capacity: int):
        self.rate = rate
        self.capacity = capacity
        self._buckets: dict[str, tuple[float, float]] = {}  # key -> (tokens, last_ts)
        self._lock = threading.Lock()

    def allow(self, key: str, cost: float = 1.0) -> Decision:
        now = time.monotonic()  # monotonic: immune to clock changes
        with self._lock:  # whole read-modify-write is atomic
            tokens, last = self._buckets.get(key, (self.capacity, now))
            # lazy refill for the elapsed interval, capped at capacity
            tokens = min(self.capacity, tokens + (now - last) * self.rate)
            if tokens >= cost:
                self._buckets[key] = (tokens - cost, now)
                return Decision(allowed=True)
            # not enough: persist the refilled state, compute wait
            self._buckets[key] = (tokens, now)
            retry_after = (cost - tokens) / self.rate
            return Decision(allowed=False, retry_after=retry_after)

# --- usage ---
# limiter = TokenBucketLimiter(rate=5.0, capacity=10)  # 5/s, burst of 10
# d = limiter.allow("key-abc")
# if not d.allowed: respond_429(retry_after=d.retry_after)

What interviewers look for.

  • The refill math is correct. tokens + elapsed × rate, clamped to capacity, computed lazily. No background timer means nothing to leak or schedule.
  • Concurrency safety. The entire read-modify-write is inside the lock. A common bug is refilling outside the lock, which double-counts under contention.
  • monotonic() not time(). Wall-clock can jump (NTP, DST) and corrupt the elapsed calculation. Using a monotonic clock is a strong senior signal.
  • Useful retry_after. Telling the caller exactly when to come back (for a 429 + Retry-After header) is the product-minded touch. Bonus: mention Redis/Lua for the distributed case and per-key bucket eviction to bound memory.

Exercise 5 — A small eval runner

Problem. Given a dataset of {input, expected} cases, a model function, and a scorer, run every case, compute the pass rate, and print a short report. Then support comparing two model variants (e.g. two prompts or two models) side by side.

Clarifying questions / constraints

  • Is the scorer exact-match or fuzzy/graded? (Make it pluggable — scorer returns a 0–1 score.)
  • Run cases concurrently with a concurrency cap? (Yes — models are I/O bound; cap to respect rate limits.)
  • Should a case that throws count as a failure or abort the run? (Count as a 0, never abort — one bad case must not sink the suite.)
  • For the comparison, do we want per-case deltas or just aggregate? (Aggregate pass rate + count of regressions.)

Hints

  • Keep three pieces independent: the dataset, the model fn, the scorer. That is what makes it reusable.
  • Wrap each case so an exception becomes a score of 0 with the error recorded.
  • For A/B, run both variants over the same dataset and diff the pass rates.
Show full solution + what interviewers look for

Python, async, with a concurrency semaphore. Model fns and scorer are plain callables, so this drops onto any real setup.

import asyncio
from dataclasses import dataclass
from typing import Awaitable, Callable

@dataclass
class Case:
    input: str
    expected: str

@dataclass
class Result:
    score: float
    error: str | None = None

ModelFn = Callable[[str], Awaitable[str]]
Scorer  = Callable[[str, str], float]  # (got, expected) -> 0..1

async def run_eval(dataset: list[Case], model: ModelFn, scorer: Scorer,
                   concurrency: int = 8) -> list[Result]:
    sem = asyncio.Semaphore(concurrency)  # respect rate limits
    async def one(c: Case) -> Result:
        async with sem:
            try:
                got = await model(c.input)
                return Result(score=scorer(got, c.expected))
            except Exception as e:  # a broken case scores 0, never aborts the run
                return Result(score=0.0, error=str(e))
    return await asyncio.gather(*(one(c) for c in dataset))

def report(name: str, results: list[Result], threshold: float = 1.0) -> float:
    n = len(results)
    passed = sum(1 for r in results if r.score >= threshold)
    errors = sum(1 for r in results if r.error)
    rate = passed / n if n else 0.0
    print(f"{name:<12} pass {passed}/{n} ({rate:.0%})  errors={errors}")
    return rate

async def compare(dataset, scorer, variant_a, variant_b):
    a = report("A: opus-4.8",   await run_eval(dataset, variant_a, scorer))
    b = report("B: sonnet-4.6", await run_eval(dataset, variant_b, scorer))
    delta = b - a
    verdict = "B wins" if delta > 0 else "A wins" if delta < 0 else "tie"
    print(f"delta {delta:+.0%} -> {verdict}")

What interviewers look for.

  • Clean separation of concerns. Dataset, model fn, and scorer are independent inputs. That is what makes the runner reusable across tasks — the core insight of an eval harness.
  • One bad case cannot sink the suite. Per-case try/except turning an exception into a 0 (with the error recorded) is exactly the robustness they probe.
  • Bounded concurrency. The semaphore shows you know model calls are I/O bound and rate-limited — naive gather over thousands of cases would get throttled.
  • A pluggable scorer + threshold. A 0–1 score with a pass threshold handles graded evals, not just exact match. The A/B comparison with a delta is the product-facing payoff. Bonus: mention seeds/determinism and saving per-case outputs for inspection.

Exercise 6 — Prompt-cache key + LRU with TTL

Problem. You want to cache responses for a stable prompt prefix (system prompt + tools + few-shot examples that rarely change). Design the cache key, then implement a small LRU cache with per-entry TTL. Demonstrate a hit and a miss.

Clarifying questions / constraints

  • What goes into the key? (Everything that changes the output: model id, the full prefix text, and relevant params like temperature.)
  • Is this the provider’s server-side prompt cache, or our own response cache? (Our own response cache — but the key design mirrors prefix-cache thinking.)
  • Eviction policy and size? (LRU with a max size, plus TTL so stale answers expire.)
  • Hash the key or store it raw? (Hash — prefixes are long; a hash keeps keys bounded and avoids logging raw prompts.)

Hints

  • Key = stable hash over (model, normalized_prefix, params). Anything that changes the answer must be in the key.
  • An ordered map gives you O(1) LRU: move-to-end on access, pop-from-front on overflow.
  • TTL: store an expiry timestamp per entry; treat expired entries as a miss and evict them.
Show full solution + what interviewers look for

Python using OrderedDict for O(1) LRU. The key builder hashes the model + prefix + params.

import hashlib, json, time
from collections import OrderedDict
from typing import Any

def cache_key(model: str, prefix: str, params: dict) -> str:
    # Everything that changes the output goes in the key.
    # Normalize prefix (strip trailing ws) and sort params for stability.
    payload = json.dumps(
        {"model": model, "prefix": prefix.strip(), "params": params},
        sort_keys=True,
    )
    return hashlib.sha256(payload.encode()).hexdigest()  # bounded, log-safe

class LRUCacheTTL:
    def __init__(self, max_size: int, ttl_seconds: float):
        self.max_size = max_size
        self.ttl = ttl_seconds
        self._store: "OrderedDict[str, tuple[Any, float]]" = OrderedDict()
        self.hits = self.misses = 0

    def get(self, key: str):
        entry = self._store.get(key)
        if entry is None:
            self.misses += 1
            return None
        value, expires_at = entry
        if time.monotonic() >= expires_at:        # expired -> miss + evict
            del self._store[key]
            self.misses += 1
            return None
        self._store.move_to_end(key)              # mark most-recently-used
        self.hits += 1
        return value

    def put(self, key: str, value: Any):
        self._store[key] = (value, time.monotonic() + self.ttl)
        self._store.move_to_end(key)
        if len(self._store) > self.max_size:
            self._store.popitem(last=False)        # evict least-recently-used

# --- demo: one miss (populates), one hit ---
cache = LRUCacheTTL(max_size=128, ttl_seconds=300)
SYSTEM = "You are a support triage assistant. ..."  # stable prefix
k = cache_key("claude-haiku-4-5", SYSTEM, {"temperature": 0})

assert cache.get(k) is None          # MISS -> call the model
cache.put(k, {"answer": "routed to billing"})
assert cache.get(k) == {"answer": "routed to billing"}  # HIT
print(f"hits={cache.hits} misses={cache.misses}")  # hits=1 misses=1

What interviewers look for.

  • Correct key design. The key includes everything that changes the answer: model id, the normalized prefix, and params like temperature. Forgetting the model id or temperature is the classic correctness bug — you would serve a Haiku answer for an Opus request.
  • Hashing the key. Prefixes are huge; a SHA-256 keeps keys bounded and avoids dumping raw prompts into logs/metrics.
  • O(1) LRU + TTL together. OrderedDict with move_to_end/popitem is the clean idiom; expired entries treated as a miss and evicted on read.
  • Hit/miss accounting. Tracking hit rate is the observability instinct — you cannot tune a cache you cannot measure. Bonus: connect this to provider prompt caching (put the stable prefix first so the server-side KV cache hits) and discuss invalidation when the system prompt changes.

How this shows up in the interview

Beyond writing the code, expect the interviewer to push on judgment. A few likely questions and crisp answers:

Likely questions

“The model is slow and the client just left — what happens?” Your abort signal propagates into the model call and you stop generating; cleanup runs in finally; you stop billing tokens. If you did not wire cancellation, say so and fix it — they are checking whether you noticed.

“The model returns garbage 1% of the time. How do you keep the product up?” Validate everything coming out of the model; retry with specific corrective feedback up to a small cap; then degrade gracefully (typed failure, fallback message, or a cheaper deterministic path) rather than throwing into the user’s face.

“This works for one server — how does it change at scale?” Move shared state (rate-limit buckets, caches) into Redis; make tool calls idempotent; add per-call timeouts and circuit breakers; cap concurrency to respect provider rate limits; and instrument hit rates, latency, and cost.

“How would you test this?” Inject a fake model so you can script malformed output, errors, slow responses, and aborts deterministically. Test the failure branches explicitly — that is the part most likely to break in production.

Use the real model names

If you reference models, use current ones and tier them on purpose: Opus 4.8 for the hardest reasoning, Sonnet 4.6 as the balanced default, Haiku 4.5 for cheap/fast/high-volume paths (classification, routing, cache-friendly calls). Saying “use the small model for the eval’s cheap cases” shows cost awareness.

Check your understanding

In the streaming endpoint, why is catching AbortError separately so important?

Because a client disconnect is an expected, normal event, not a server fault. If you let the abort fall into your generic error handler, you will emit spurious 500s / error events, pollute your error metrics, and possibly alert on-call for something that is just a user closing a tab. Treating AbortError as a clean exit (and still running cleanup in finally) is the correct, production-minded behavior.

Your structured-output retry loop sometimes never converges. What are two likely causes and fixes?

(1) Vague corrective messages. If you just say “that was wrong, try again,” the model has no signal to fix the specific defect. Echo the exact validation error and restate the schema. (2) Too much randomness. Retrying at the same high temperature reproduces the same class of mistake; lower the temperature on retry. Always keep a hard retry cap and a graceful give-up so a stubborn case cannot loop forever or run up cost.

Why use time.monotonic() instead of time.time() in both the rate limiter and the cache?

Both rely on measuring elapsed time, and the wall clock can jump backward or forward (NTP corrections, daylight-saving, manual changes). A backward jump could make the rate limiter compute negative elapsed time (refill nothing or break) or make a TTL never expire. monotonic() only ever moves forward at a steady rate, so it is the correct clock for durations. Use time.time() only when you need an actual calendar timestamp.

What is the single most common reason a prompt-response cache returns a wrong answer?

An incomplete cache key. If the key omits something that changes the output — the model id, the temperature, a tool definition, or part of the prefix — you will serve a response generated under different conditions. The discipline is: every input that can change the answer must be in the key. When in doubt, hash more, not less; the cost of an over-specific key is a missed hit, while the cost of an under-specific key is a silently wrong answer.
