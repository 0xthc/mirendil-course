KV cache & prompt caching (deep)

Two things share the word “cache” and get conflated constantly, including in interviews. The KV cache is the engine’s private scratchpad inside one request. Prompt caching is an API feature you control to reuse work across requests. They are the same kind of object — a saved prefill state — living on two different timescales. This page pulls them fully apart, then shows exactly how they connect.

ELI5

Imagine the model is a chef. To cook your dish it first reads the whole recipe and lays out every ingredient on the counter — that prep is the KV cache: it exists only while this one dish is being cooked, and it gets thrown out when the plate goes out. Now imagine ten customers all order the same base sauce. Prompt caching is the chef saying “keep that base sauce warm on the side” so the next order skips the prep entirely. Same prepped ingredients — one is for the dish in front of you, the other is kept around for the next customer.

The one table that disambiguates them

If you remember nothing else, remember this contrast. Almost every confusion about caching dissolves once you separate within one request from across requests.

The punchline

Prompt caching is literally “save the KV cache produced by a previous request’s prefill, and reuse it.” The KV cache is the object; prompt caching is the policy that lets that object outlive the one request that built it. Different lifetime, same thing.

Part 1 — The KV cache (the mechanism)

Why it has to exist

Generation is sequential: the model writes one output token, then the next, each one a full forward pass. To produce a token, the attention mechanism has every position’s query attend against the keys and values of all earlier tokens. Here is the crucial observation: when you append a new token, the keys and values of all the earlier tokens do not change. They are a function of tokens that are already fixed. So recomputing them on every step would redo the same work over and over — O(n²) per layer across the whole generation.

The fix is to compute each token’s K and V exactly once and keep them. After a token is processed, its keys and values are written into the KV cache. Each new step then computes only the K/V for the one new token and attends against everything already saved. That turns the per-token cost from “reprocess the whole sequence” into “process one token,” making total decode O(n) instead of O(n²).

How it ties to prefill vs decode

This maps cleanly onto the two phases of a request:

  • Prefill reads the prompt and populates the cache. All prompt tokens go through in parallel, so it is compute-bound and fast per token.
  • Decode generates the answer and reads the whole cache on every single output token. Each step reads all the weights plus the entire KV cache out of memory to produce one token, so it is memory-bandwidth-bound.

That asymmetry is exactly why output tokens dominate latency: each one is a separate sequential pass that drags the full, growing cache through memory.

How big is it? The size math

The cache stores a key and a value for every token, at every layer, for every KV head. The formula:

kv_bytes = 2 × n_layers × n_kv_heads × head_dim × seq_len × bytes_per_value

The leading 2 is for storing both K and V. bytes_per_value is 2 for bf16.

Worked example. Take a model with 32 layers, 8 KV heads, head_dim 128, in bf16:

2 × 32 × 8 × 128 × 2 bytes = 131,072 bytes per token ≈ 131 KB/token

× 8,000 tokens ≈ 1 GB  ·  × 128,000 tokens ≈ 16 GB

And that is for ONE request. A single 128K-context conversation can hold ~16 GB of KV cache resident on the GPU for its entire lifetime.

Why it bounds throughput

GPU memory is finite, and the KV cache competes with the model weights for it. Whatever is left after the weights determines how many requests’ caches can be resident at once — and that is your batch size, which is your throughput. Long context multiplied by batch size caps how many concurrent requests you can serve.

This is why GQA / MQA matter so much. They shrink n_kv_heads: many query heads share a smaller set of KV heads. A model with 32 query heads but only 8 KV heads has a cache about 4× smaller than if every query head had its own KV head — directly buying back batch size and longer usable context.

Gotcha

“Long context” is not just a token bill. It is expensive three ways at once: compute (more to prefill), memory (a bigger KV cache resident for the whole request), and throughput (that memory is no longer available to hold anyone else’s request). When you double a context window you are quietly deciding how many other users you can serve.

Part 2 — Prompt caching (the API feature)

Now the thing you actually control. Prompt caching persists the prefill KV state of a stable prefix so that later requests reuse it instead of recomputing it — much cheaper, and faster TTFT.

How you turn it on

Add cache_control to a content block. The block (and everything before it) becomes a cacheable prefix:

  • {"type":"ephemeral"} — 5-minute TTL (the default), refreshed on every hit, so a busy prefix stays warm.
  • {"type":"ephemeral","ttl":"1h"} — 1-hour TTL.
The one rule that governs everything

Prompt caching is a prefix match. Any byte change anywhere in the prefix invalidates the cache for everything after that point. The render order is tools → system → messages. So put stable content first and volatile content (timestamps, the user’s question, per-request IDs) last. Order is the whole game.

The numbers that matter

FactValue
Breakpoints per requestMax 4 (on system / tool / message / document / image blocks).

Below the minimumIt silently does not cache — no error.

Why the write premium still pays off

The first request writes the cache at 1.25×, so a single call is slightly more expensive than not caching. Every later call with the same prefix reads at ~0.1×. On a large shared prefix the crossover comes almost immediately — by the second request you are already ahead.

Verify it actually works

The response usage object tells you exactly what happened. Three fields:

  • cache_creation_input_tokens — tokens just written to the cache (billed at the write rate).
  • cache_read_input_tokens — tokens served from the cache (billed ~0.1×).
  • input_tokens — ordinary uncached input.

If cache_read_input_tokens stays 0 across requests that should share a prefix, you have a silent invalidator somewhere in that prefix. That number is your single best debugging signal.

import anthropic
client = anthropic.Anthropic()

STABLE = "...your long, byte-stable system prompt (>= the model minimum)..."

resp = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=512,
    system=[{
        "type": "text",
        "text": STABLE,
        "cache_control": {"type": "ephemeral"},   # 5-min TTL breakpoint
    }],
    messages=[{"role": "user", "content": user_question}],  # volatile: goes last
)

# First call writes; identical-prefix calls after it read.
print(resp.usage.cache_creation_input_tokens)  # > 0 on the write
print(resp.usage.cache_read_input_tokens)      # > 0 on every hit

The silent killers

Gotcha

Most “prompt caching does nothing” reports are a volatile byte sitting inside the prefix. The classic culprit is a datetime.now() or a fresh uuid in the system prompt — it changes every request, so the prefix never matches and you re-prefill every time. Watch cache_read_input_tokens: if it is stuck at 0, hunt for the moving byte.

MistakeWhat happensFix
Timestamp / UUID / request-id in the system promptPrefix differs every call → 0 hits.Move volatile values to the end, after the breakpoint.
Non-deterministic JSON (no sort_keys, iterating a set)Byte order varies → prefix mismatch.Serialize deterministically (sorted keys, ordered structures).
Changing tools mid-conversationTools render at position 0 → invalidates everything.Keep the tool set fixed for the life of the cache.
Switching the model mid-conversationCaches are model-scoped → full miss.Pin one model per cached conversation.
Toggling tool_choice or thinkingDoes NOT drop the tools+system cache (tiered invalidation).Safe — these are fine to vary.
Very long agentic turnsOnly a ~20-block lookback window is checked for cache hits.Add an intermediate breakpoint every ~15 blocks.
Concurrent fan-out from a cold cacheAn entry is only readable after the first response begins streaming.Fire one request, await its first token, then fan out the rest.

Where it wins

  • A long, stable system prompt shared across many users — cache it once, every user reads it.
  • One long document, many questions — document before the breakpoint, the question after it.
  • RAG context that is reused across a session.
  • Multi-turn chat — put the breakpoint on the last turn so the growing history stays cached.
  • Agent loops — the identical tools + system prompt are re-sent on every step; caching makes each step cheap.

Part 3 — How they connect

Now the whole picture in one frame. They are the same object on two timescales:

KV cachePrompt caching
TimescaleWithin one request.Across requests.
What it buysO(n²) → O(n) decode.Skip re-prefilling a stable prefix (~0.1× to reuse).
ControlAutomatic, invisible.You, via cache_control; TTL’d.
The objectThe same prefill KV state — prompt caching just persists it past the one request that built it.

So: the engine always builds a KV cache to make a single request fast. Prompt caching is you telling the provider “don’t throw that prefill away when this request ends — keep it for the next one that starts with the same bytes.”

How this shows up in the interview

This is a favorite because it separates people who have shipped on models from people who have only read about them. Expect these.

Q: Why is long context expensive?

Three mechanisms, all flowing from the KV cache. Compute: a longer prompt is more prefill work. Memory: the KV cache grows linearly with context (~131 KB/token for a 32-layer/8-KV-head/bf16 model — ~16 GB at 128K) and is resident for the whole request. Throughput: that resident memory competes with the weights and with every other request’s cache, so long context shrinks batch size and the number of concurrent users you can serve. Long context costs latency and memory and who else you can serve.

Q: Cut cost on a chatbot with a 10K-token system prompt.

Prompt caching, breakpoint on the system block. The 10K tokens easily clear every model minimum, so put a cache_control ephemeral breakpoint there, keep that prefix byte-stable (no timestamps/UUIDs — those go after the breakpoint), and pin one model. Then verify: watch cache_read_input_tokens climb on repeat calls. Cached reads at ~0.1× mean the system prompt goes from full price every turn to roughly a tenth, with break-even after about the second request. If cache_read stays 0, a volatile byte slipped into the prefix.

Q: KV cache vs prompt caching — one line each.

KV cache: the engine’s automatic, per-request scratchpad that makes decode O(n) instead of O(n²). Prompt caching: an API feature that persists a request’s prefill KV state for a stable prefix so later requests reuse it at ~0.1×. Same object, different lifetime.

Check your understanding

You enable prompt caching on a 6K-token system prompt for an Opus 4.8 chatbot, but cache_read_input_tokens is always 0. The prefix is above the 4096-token minimum. What is most likely wrong?

A volatile byte inside the prefix. Prompt caching is a prefix match, so anything that changes per request — a datetime.now(), a session UUID, a request id, or non-deterministically serialized JSON — invalidates everything after it, and the cache never gets a hit. Move all volatile content after the breakpoint (render order is tools → system → messages), serialize JSON deterministically, and confirm the model and tool set are not changing mid-conversation. The 0 in cache_read is the tell.

A model has 32 query heads but only 8 KV heads (GQA). How does that change the KV cache, and why does it matter for serving?

The cache size depends on n_kv_heads, not n_query_heads. With 8 KV heads instead of 32, the cache is about 4× smaller than full multi-head attention would be. Since the KV cache competes with the weights for GPU memory and caps how many requests fit in a batch, a 4× smaller cache directly buys back batch size (throughput) and longer usable context. That is the entire point of GQA/MQA.

For a single one-off API call with a large prefix, prompt caching makes it slightly more expensive, not cheaper. Why — and when does it flip?

The first call has to write the cache, billed at 1.25× base input (for the 5-min TTL), so a lone request pays a premium over not caching at all. The savings come from later reads at ~0.1×. With a large shared prefix the crossover lands at about the second request (5-min TTL; ~3 for the 1-hour TTL). One-and-done calls should not bother; anything repeated should.
