Talking to models in production

A model in a product is just an HTTP endpoint you POST a list of messages to. That sounds simple, and the happy path is. The job — the part that separates a demo from a product — is everything around it: shaping the conversation, getting structured data back reliably, letting the model call your code, streaming tokens so the UI feels alive, and handling the dozen ways a network call can go sideways. This page is the day-to-day API mechanics a product engineer touches constantly.

ELI5

Talking to a model is like texting a very capable contractor who has no memory between texts. Every time you message them, you must re-send the whole thread so they remember what you are doing. You also hand them a job description up front (the system prompt), you can give them tools and say here is how to phone the warehouse if you need a part (tool calling), and you can ask them to reply on a strict form instead of free prose (structured output). The contractor is brilliant but literal — so the shape of how you ask matters as much as what you ask.

The shape of a request: a list of messages

Almost every chat-style model takes the same core input: an ordered list of messages, each tagged with a role. You send the whole list every turn — the model is stateless, so the conversation lives in your array, not on the server.

messages = [
  {"role": "system",    "content": "You are a terse support agent for Acme..."},
  {"role": "user",      "content": "My order never arrived."},
  {"role": "assistant", "content": "I can help. What is your order number?"},
  {"role": "user",      "content": "A-10473"}
]

The three roles, and what belongs where

RoleWho it representsWhat goes in it
systemYou, the productDurable instructions: persona, rules, format, tone, what the model may/may not do, available context. Set once, applies to the whole conversation.
userThe end user (or your app on their behalf)The actual request or turn. Also where you inject retrieved documents, the user's selection, form data.
assistantThe modelThe model's prior replies. You echo them back so the model sees its own history. You can also prefill a partial assistant turn to steer the start of its answer.

On the Claude API the system prompt is a top-level system field rather than a message in the array, but the mental model is identical: stable instructions live separate from the turn-by-turn dialogue.

The system prompt is your product spec, not a greeting

Put everything durable and trusted in the system prompt: role, hard rules, output format, refusal policy, tool-usage guidance. Put everything per-turn and user-controlled in user messages. This split matters for safety too — the model is trained to weight system instructions above user instructions, so a rule like never reveal account balances belongs in system, where a user message saying ignore your rules has less leverage against it.

Few-shot examples: teach by showing

When a rule is hard to state but easy to demonstrate, give the model worked examples as fake prior turns. A handful of user → assistant pairs showing the exact input/output behavior you want often beats paragraphs of instructions. This is few-shot prompting, and it is the cheapest reliability lever you have before reaching for fine-tuning.

messages = [
  {"role": "system", "content": "Classify each message as BILLING, BUG, or OTHER. Reply with one word."},
  {"role": "user", "content": "I was charged twice"},        # example 1
  {"role": "assistant", "content": "BILLING"},
  {"role": "user", "content": "the app crashes on launch"},  # example 2
  {"role": "assistant", "content": "BUG"},
  {"role": "user", "content": "do you ship to Canada?"}      # the real one
]
ELI5 — few-shot

Instead of explaining the rules of a card game, you play two hands face-up while the new player watches, then deal them in. They pattern-match off what they just saw. Models do the same: two or three clean examples pin down the format and edge cases far better than a wall of prose.

Structured outputs: getting reliable JSON

Most product features do not want prose — they want data: a category, a list of extracted fields, a function argument. The naive move is to add respond in JSON to the prompt. That works in the demo and fails in production, because plain text generation has no guarantee of being valid JSON.

Why “just ask for JSON” is fragile

A free-text model will, often enough to hurt, wrap the JSON in a markdown code fence, add a chatty preamble like Sure, here is the JSON:, emit a trailing comma, use single quotes, hallucinate an extra field, or get truncated mid-object when it hits the token limit. At 99% success that is one broken response in a hundred — a constant stream of 3am pages at any real volume. You need a mechanism that constrains the structure, not a polite request that hopes for one.

Two reliable mechanisms

Modern APIs give you two ways to enforce structure, and they share the same engine under the hood (constrained decoding against a schema):

  • Response schema / JSON mode — you attach a JSON Schema to the request; the API constrains generation so the output validates against it. Best when you want one structured object as the answer.
  • Tool / function calling — you define a tool whose input schema is your desired shape, and force the model to call it. The tool-call arguments come back as schema-valid JSON. This is the most portable trick: even APIs without a dedicated JSON mode support tools, so “define a tool you never actually run, just to extract fields” is a common pattern.

Here is a concrete JSON Schema for an extraction task — pulling a support ticket into fields:

{
  "type": "object",
  "properties": {
    "category": { "type": "string", "enum": ["billing", "bug", "other"] },
    "urgency":  { "type": "integer", "minimum": 1, "maximum": 5 },
    "summary":  { "type": "string", "maxLength": 280 },
    "needs_human": { "type": "boolean" }
  },
  "required": ["category", "urgency", "summary", "needs_human"],
  "additionalProperties": false
}

And the same shape expressed as a Claude tool definition — note the input_schema is the JSON Schema above:

tools = [{
  "name": "record_ticket",
  "description": "Save the structured support ticket. Always call this exactly once.",
  "input_schema": {
    "type": "object",
    "properties": {
      "category": { "type": "string", "enum": ["billing", "bug", "other"] },
      "urgency":  { "type": "integer", "minimum": 1, "maximum": 5 },
      "summary":  { "type": "string" }
    },
    "required": ["category", "urgency", "summary"]
  }
}]

# Force the model to use it, so you always get structured args back:
resp = client.messages.create(
    model="claude-sonnet-4-6",
    tools=tools,
    tool_choice={"type": "tool", "name": "record_ticket"},
    messages=messages,
)

Validate, then retry — even with schema enforcement

Constrained decoding makes malformed JSON rare, not impossible (truncation at max_tokens can still cut an object in half). So always parse and validate against your schema in code, and on failure, retry with the error fed back in:

def extract(messages, attempts=3):
    for i in range(attempts):
        resp = call_model(messages)
        try:
            data = json.loads(resp.text)
            schema.validate(data)          # jsonschema / pydantic / zod
            return data                    # success
        except (ValueError, SchemaError) as e:
            messages += [
              {"role": "assistant", "content": resp.text},
              {"role": "user",
               "content": f"That did not validate: {e}. Return ONLY valid JSON for the schema."}
            ]
    raise ExtractionFailed(messages)    # give up loudly, do not silently swallow
The reliability stack, in order

(1) Constrain structure with a schema or forced tool call — do not rely on prose instructions. (2) Parse and validate in your own code; the API contract is bytes, not meaning. (3) Retry with the validation error fed back, capped at a small number of attempts. (4) Fail loudly when retries exhaust — never return a half-parsed object downstream. Each layer catches what the one above misses.

Tool / function calling: the round trip

Tool calling is how a model reaches outside its own text box — to look up an order, query a database, hit a weather API, run a calculation. The crucial thing to understand is that the model never runs your code. It only emits a request to run a tool; your application runs it and feeds the result back. The loop is:

Concretely, after the model returns a tool call, your next request appends two messages — the assistant's tool-use turn and your tool-result turn:

# ... earlier messages ...
{"role": "assistant", "content": [
   {"type": "tool_use", "id": "tu_01",
    "name": "get_order_status", "input": {"id": "A-10473"}}
]},
{"role": "user", "content": [
   {"type": "tool_result", "tool_use_id": "tu_01",
    "content": "{\\"status\\": \\"shipped\\", \\"eta\\": \\"Tue\\"}"}
]}
# then call the API again with this extended thread
The result must be tied back by id

Each tool_result carries the tool_use_id of the call it answers. If the model fires several tool calls in one turn (parallel tool use), you must return a result block for every id before the model will continue — miss one and the next request errors. Also: never trust tool arguments blindly. The model can hallucinate an id or pass an out-of-range value, so validate and authorize inside your tool exactly as you would for untrusted user input.

Streaming and cancellation

A model generates one token at a time. If you wait for the whole response, the user stares at a spinner for several seconds. If you stream, you forward each token as it is produced and the answer appears to type itself — the total time is identical, but perceived latency collapses because time-to-first-token is a few hundred milliseconds instead of the full generation.

Mechanically, instead of one JSON response you open a stream (server-sent events) and receive a series of small events — a start, many incremental token deltas, then a stop. You append deltas to the UI as they arrive:

const stream = await client.messages.stream({
  model: "claude-haiku-4-5",
  messages,
});
for await (const event of stream) {
  if (event.type === "content_block_delta")
    ui.append(event.delta.text);   # paint each token as it lands
}

Cancellation: stop paying for tokens nobody wants

Because the model bills per output token, a generation the user has abandoned — they closed the tab, hit stop, or edited their question — is pure waste. The fix is an abort signal: a handle you can trip to tear down the in-flight request. The server stops generating, and you stop being billed for the remaining tokens.

const controller = new AbortController();
# Pass the signal into the SDK / fetch call:
const stream = client.messages.stream(
  { model: "claude-sonnet-4-6", messages },
  { signal: controller.signal }
);
# Later, when the user clicks Stop or the request times out:
stopButton.onclick = () => controller.abort();
Streaming is UX; cancellation is cost

Two separate wins from the same plumbing. Streaming buys you perceived latency — the answer feels instant even though it is not. Cancellation buys you money and capacity back — abandoned generations stop consuming output tokens and rate-limit budget. Ship streaming for the feel; wire the abort signal so the feel does not cost you on every dropped request.

The unglamorous reliability details

Determinism and temperature

Temperature controls randomness in sampling. At temperature = 0 the model picks the highest-probability token at each step, which is as deterministic as it gets — best for extraction, classification, and anything you will parse. Higher values (toward 1.0) add variety, useful for brainstorming or copy. Note even at 0 you are not guaranteed byte-identical output across calls — floating-point and infrastructure nondeterminism can still cause rare divergence — so do not build a system that assumes the same prompt always yields the same bytes.

Idempotency and retries on network errors

A timeout or 5xx does not tell you whether the request was processed — the response may have been generated and lost on the way back. Blind retries can double-bill or double-act (imagine a tool that charges a card). Use an idempotency key so a retried request is recognized as the same one, and retry with exponential backoff plus jitter on 429 (rate limit) and 5xx, but not on 4xx like a malformed request, which will fail identically every time.

Truncation: hitting max_tokens

Every response has a max_tokens cap. If the model hits it mid-sentence, you get a partial answer and a stop_reason of max_tokens (not end_turn). Always check the stop reason: if it is max_tokens, the output is truncated — your JSON may be half an object. Handle it by raising the cap, asking the model to continue, or treating it as a validation failure and retrying.

Refusals

The model may decline a request on safety grounds, returning a polite refusal as normal assistant text (and on some APIs a distinct stop_reason such as refusal). Your parser must not mistake a refusal for a malformed structured output and retry forever — detect it, and route to a human or a fallback rather than hammering the API.

Always read stop_reason before you trust the body

The single most common production bug here: code that grabs response.content and parses it without checking why the model stopped. end_turn means a complete answer; tool_use means it wants you to run a tool; max_tokens means it is truncated; refusal means it declined. Branch on the stop reason first — half of robust model plumbing is just respecting that field.

How this shows up in the interview

“Design tool calling for an agent that can look up and refund orders.”

Walk the round trip explicitly: define tools with clear names, descriptions, and input_schema; send them with the conversation; the model returns a tool_use block with arguments; your code executes the real function; you append the tool_use assistant turn plus a tool_result keyed by tool_use_id; call again; loop until stop_reason is end_turn. Then show seniority: the model only proposes, so authorization and validation live in your tool, not the prompt — a refund tool checks the caller actually owns the order and enforces limits server-side. Add an idempotency key so a retried refund does not double-charge, and cap the loop length to prevent runaway tool cycles.

“How do you make structured output reliable?”

Lead with why prose instructions fail: free-text generation has no structural guarantee, so you get fences, preambles, trailing commas, and truncation. Then give the stack — constrain with a JSON Schema / forced tool call (constrained decoding), parse and validate in your own code, retry feeding the validation error back, and fail loudly after a small cap. Mention that even constrained output can be truncated at max_tokens, so you still check stop_reason. The signal they want: you treat the API contract as bytes you must validate, not meaning you can assume.

“Implement streaming with cancellation.”

Separate the two wins: streaming is for perceived latency (forward token deltas over SSE; time-to-first-token drops to a few hundred ms while total time is unchanged), cancellation is for cost (an AbortController whose signal is threaded into the SDK call, tripped on user-stop, tab-close, or timeout — the server halts generation and billing stops). Mention the failure mode of not doing it: abandoned generations keep consuming output tokens and rate-limit budget. Bonus points for handling partial state cleanup and for noting that a cancel still bills for tokens already emitted.

Check your understanding

A teammate adds “respond only in JSON” to the system prompt and ships it. It works in testing. Why will it page someone, and what do you change?

Plain-text generation has no structural guarantee, so at volume you will get markdown fences, chatty preambles, trailing commas, or output truncated at max_tokens — each a parse failure. Replace the polite request with an enforced mechanism: a response JSON Schema or a forced tool call (constrained decoding), then validate the result in code and retry with the error fed back, capping attempts and failing loudly. The prompt asks; the schema enforces.

The model returns a tool_use block. Did it run your function? What exactly do you send on the next request?

No — the model never executes anything; it only emits a request to run the tool, with arguments. Your code runs the real function (validating and authorizing the arguments first). On the next request you append two things to the thread: the assistant turn containing the tool_use block, and a tool_result message carrying the return value and the matching tool_use_id. Then you call the API again; the model uses the result to continue or to call another tool.

A user closes the tab two seconds into a long streamed answer. With and without an abort signal, what happens to your bill?

Without cancellation, the server keeps generating to completion and you are billed for every output token, even though nobody will see them — plus it consumes your rate-limit budget. With an AbortController whose signal is passed into the SDK call, tripping controller.abort() tears down the request, the server halts generation, and you only pay for the tokens emitted before the abort. On long abandoned answers that is most of the cost saved.

Your extraction call returns content that fails JSON parsing. Before retrying, which one field should you check, and why?

Check stop_reason. If it is max_tokens, the output is truncated — the JSON is incomplete because generation was cut off, so the fix is to raise the cap or ask it to continue, not to re-prompt for valid JSON. If it is refusal, the model declined, and retrying forever just burns money — route to a human or fallback. Only if it stopped normally (end_turn) does a “that did not validate, try again” retry make sense.
