Module 07

Tensor-sequence parallelism (TSP)

The paper's contribution and the heart of the test. Fold both the weight split and the token split onto the same GPUs, so a single 8-GPU group shrinks model-state and activation memory together.

TSP vs "TP + SP" — get this distinction first

There are two ways to combine the two ideas, and they are not the same. This caused real confusion in the sessions, so we'll be explicit.

TP + SP: a 2D grid of GPUs (two separate axes)

Give yourself T × Σ GPUs arranged in a grid. One axis does tensor parallelism, the other does sequence parallelism. With T=2, Σ=2 you need 4 GPUs, and every (head-shard, token-shard) combination physically exists:

head shard 1
seq shard 0

rank 1

TP+SP, 4 GPUs. Every grid cell is filled. No forward-time weight movement is needed — each rank permanently holds its (head, token) combination.

TSP: fold both onto one axis (the diagonal)

TSP is more economical with GPUs: it puts both roles on the same set of GPUs. With D=2 ranks, rank i owns sequence-shard i and head-shard i — the diagonal of that grid. The off-diagonal cells don't physically exist anywhere:

head shard 0
head shard 1
seq shard 0
rank 0
missing
seq shard 1
missing
rank 1
TSP, 2 GPUs. Only the diagonal exists. Rank 0 has (seq 0, head 0); rank 1 has (seq 1, head 1). The "missing" cells are why TSP needs a loop.
The defining problem of TSP

Rank 0 holds sequence-shard 0 but only head-shard 0's weights. To produce the full output for its tokens, rank 0 needs every head shard applied to its tokens — including head shards it doesn't own. So TSP loops over all weight (head) shards, circulating each shard to every rank in turn. That loop is the cost of folding two axes onto one set of GPUs.

Algorithm 1: TSP attention, line by line

This is the implementation target from the paper. r is the weight-owner index (which head shard we're currently processing); p is this rank's sequence-shard. We accumulate the output for our local tokens, Y_p, across all weight shards.

Y_p = 0
for r in 0..D-1:                          # loop over every head/weight shard
    broadcast [Wq_r, Wk_r, Wv_r, Wo_r]      # circulate shard r's weights to all ranks
    Q_rp = X_p @ Wq_r.T                      # my tokens, shard r's query proj
    K_rp = X_p @ Wk_r.T
    V_rp = X_p @ Wv_r.T
    K_r, V_r = all_gather(K_rp, V_rp)        # gather K/V over SEQUENCE ranks p
    A_rp = causal_attn(Q_rp, K_r, V_r)       # my tokens attend to full sequence, shard r
    Y_p += A_rp @ Wo_r.T                      # accumulate shard r's residual contribution

Wq_r, Wk_r, Wv_r[H/D, H]shard r's Q/K/V weights (one head group)

The two "aha"s that took real correction

1. There is NO final all_reduce

You might expect, like TP, an all_reduce at the end. There isn't one — and this was an explicit correction in the sessions. Why not? Each rank owns a different set of output tokens (it's sequence-sharded). So there's nothing to sum across ranks. The sum that does happen — summing over head shards — happens locally, as the Y_p += … accumulation inside the loop. Rank p's loop walks every head shard and adds up their contributions to its own tokens. The cross-rank reduction TP needed is replaced by a local sum over loop iterations.

Two other corrections from the notes, so you don't repeat them

• Include Wo in the broadcast — it's easy to forget the output-projection shard and only circulate Q/K/V.
• Use causal_attn(Q, K, V) — not causal_attn(Q, K, K). A genuine typo-class bug: the value argument must be V, not K.
• For a fixed weight shard r, the all_gather is over the sequence ranks p — you're rebuilding the full key/value sequence, just as in SP.

2. Why the loop exists at all (the 2-GPU teaching version)

On only 2 ranks, the loop is compensating for the missing off-diagonal grid cells. Rank 0 has (seq 0, head 0) and rank 1 has (seq 1, head 1); the pairs (seq 0, head 1) and (seq 1, head 0) exist nowhere. The loop broadcasts each head shard to every sequence shard so the missing combinations get computed on the fly. As the session put it: "The loop is not accidental; it is compensating for the missing grid cells."

Try it: the shape calculator

Look at the TSP rows. X_p is [B, S/D, H] (activations shrank, like SP) and each weight shard is [H/D, H] (weights shrank, like TP). That simultaneous reduction on one set of GPUs is the entire promise of TSP.

TSP for the MLP

Same spirit: weight shards rotate across ranks while each rank accumulates the output for its local tokens. Because the MLP is token-independent there's no K/V gather, but there's still the weight movement — each rank needs every weight shard applied to its tokens, so the shards circulate and outputs accumulate locally.

The honest caveat the report makes

This teaching implementation is deliberately slow

Broadcasting weight shards during every forward pass is expensive, and the loop means each rank touches every head shard (so it isn't getting TP's compute savings — only the memory savings). The report is upfront: real implementations would keep shards resident, overlap the weight movement with compute, and use ring-attention instead of full K/V gather. The version in the repo prioritizes correctness and clarity so you can see the algorithm. Module 8 quantifies exactly what that costs.

Check your understanding

With D=2, why does rank 0 need a loop at all — doesn't it have everything for its own tokens?

No. Rank 0 has its tokens (seq shard 0) but only head shard 0's weights. The full output for its tokens needs all head shards applied to them, including head shard 1, which lives on rank 1. The loop circulates each head shard to rank 0 (and every rank) so the missing (seq 0, head 1) combination gets computed. On the diagonal-only layout, the off-diagonal cells must be reconstructed by moving weights.

TP attention ends with all_reduce. Why does TSP attention NOT, even though it also sums over head shards?

In TP the sum over head shards is spread across ranks (each rank holds different heads of the same tokens), so it needs a cross-rank all_reduce. In TSP each rank already iterates over every head shard itself, inside its loop, and accumulates into Y_p locally. The head-shard sum is a local +=, not a collective. And because each rank owns disjoint output tokens (sequence-sharded), there's nothing to reduce across ranks at all.

What's the difference between "TSP" and "TP+SP", in one line each?

TP+SP: a 2D grid of T×Σ GPUs; every (head, token) combination physically exists; no forward-time weight movement. TSP: both roles folded onto the same D GPUs (the diagonal); off-diagonal combinations are reconstructed by circulating weight shards in a loop during the forward pass. TSP uses fewer GPUs for the same two reductions but pays in weight traffic.
