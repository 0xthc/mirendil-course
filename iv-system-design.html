Interview prep

System-design walkthroughs

A reusable method for LLM system-design interviews, then four worked walkthroughs you can study end to end. The goal is not to memorize answers — it is to internalize a structure you can apply to any prompt the interviewer throws at you.

ELI5

A system-design interview is like being asked to plan a road trip out loud. Nobody expects you to already know every street. They want to see how you think: where are we going (requirements), what does the car need to do (the contract), what is the route (data flow), what happens at rush hour (scale), what if a tire blows (failure), and what does gas cost (budget). Say those six things in order, out loud, every time — and you will sound senior even on a topic you have never seen.

What is actually being graded

Interviewers are grading your structure and your tradeoff reasoning — not whether you arrive at one canonical right answer. A candidate who names the constraints, proposes a clean contract, and says here is the cheap option, here is the fast option, and here is why I would pick this one beats a candidate who blurts a perfect-sounding architecture with no justification. Think out loud. Make your assumptions explicit. Compare at least two options at every fork.

Part A · A method for LLM system-design interviews

Run these six steps in order, every time. The first two are where most candidates skip ahead and lose points; slow down there. The last two are where senior candidates separate themselves; do not run out of time before you reach them.

  1. Clarify requirements & scale. Who uses this, for what? Read-heavy or write-heavy? What is the QPS today and at 100×? Latency target (interactive chat vs. batch)? What is the context size — a 2K prompt or a 200K-token document? What is explicitly out of scope? Write the numbers down where the interviewer can see them.
  2. Sketch the API / contract. Define the request and response shape before any boxes. One or two endpoints, the fields, whether it streams, and the error contract. The contract pins down the problem and stops you from designing in the air.
  3. Data flow & components. Draw the path a request takes through the system, box by box: gateway, the LLM call, any retrieval, any cache, the datastore, the queue. Keep it to the components the contract actually needs.
  4. Scale it. Now stress the diagram: where is the latency budget spent, what is the throughput ceiling, what do you cache, what do you replicate, what do you shard. Tie every choice back to a number from step 1.
  5. Failure modes & reliability. What happens when the model provider is slow or returns a 529, when a dependency is down, when you hit a rate limit, when a request is retried? Timeouts, retries with backoff, idempotency, graceful degradation, fallback to a smaller model.
  6. Cost. Put a dollar figure on it. Tokens dominate LLM cost, so estimate input/output tokens per request × price per token × volume, then name the biggest lever (usually prompt caching, model tiering, or shorter context).
The most common way candidates lose

Diving into code or component boxes before clarifying requirements. The interviewer says build a rate limiter and the candidate immediately starts writing a token-bucket class. Stop. You do not yet know: limit per user or per org? Limit requests, tokens, or both? What is the QPS? Single region or global? Five clarifying questions take ninety seconds and reshape the entire design. Asking them is not a stall — it is the single strongest signal of seniority you can send.

Part B · Four worked walkthroughs

Each walkthrough follows the same skeleton: the prompt, clarifying questions, a proposed design with a request-path flow and a static architecture sketch, key tradeoffs, scaling / failure / cost notes, and the follow-ups an interviewer is likely to push on.

1 · A prompt-caching layer for a chat product

Prompt: Our chat product re-sends a large system prompt and growing conversation history on every turn. Design a caching layer that cuts latency and cost.

Clarifying questions

  • Is the cost driver a long, stable system prompt / tool schema, a long document in context, or the conversation history itself? (Changes what is cacheable.)
  • Are we talking about provider-side prompt caching (the model vendor caches the KV state of a prefix) or an application-side response cache (we skip the model entirely for repeated inputs)? Both are valid; they solve different problems.
  • How long do conversations live, and how many turns? What is the cache TTL we can tolerate?
  • What is the target hit rate, and what is the penalty for a stale answer?

Proposed design

Two layers, because they win in different places:

  • Provider prompt caching for the stable prefix. Order the prompt so the unchanging parts come first — system prompt, tool definitions, few-shot examples, any pinned document — and mark a cache breakpoint after them. The provider caches the KV state of that prefix; subsequent turns reuse it. Cached input tokens are far cheaper and skip re-prefill, which cuts time-to-first-token. This does not change the output, so there is no correctness risk.
  • Application response cache (Redis) for exact or near-exact repeated requests — FAQ-style turns, regenerations, idempotent retries. Key on a hash of the full resolved prompt; return the stored completion on a hit and never call the model.

client
  │ chat turn
  ▼
[ API gateway ] ──▶ [ prompt assembler ]  stable prefix first, volatile suffix last
  │
  ▼
[ response cache · Redis ] ──hit──▶ return stored completion
  │ miss
  ▼
[ model client ] ──▶ [ LLM w/ prompt caching ]  prefix served from KV cache
  │
  ▼
write-through to Redis (TTL)  +  stream to client

Two caches in series: an application response cache that can skip the model, and provider prompt caching that discounts the calls you still have to make.

Key tradeoffs

  • Prefix ordering vs. flexibility. Prompt caching only helps if the prefix is byte-identical across turns. Any per-turn personalization injected near the top busts the cache. You trade some prompt flexibility for cache hits — keep volatile content at the end.
  • Response cache freshness vs. hit rate. A long TTL lifts hit rate but risks serving stale answers; for anything time-sensitive or user-specific, scope the key tightly or skip layer 1 entirely.
  • Cache-write cost. Provider prompt caching typically charges a small premium to write the cache and a deep discount to read it, so it pays off only when the prefix is reused enough times to amortize the write.

Scaling, failure & cost

Hit-rate math. Say a turn is 8,000 stable-prefix tokens + 500 fresh tokens, and a conversation averages 10 turns. Without prompt caching you pay full input price on 8,500 tokens × 10 = 85,000 input tokens. With prompt caching, you pay full price for the prefix once (turn 1), then cached-read price on the 8,000-token prefix for the other 9 turns. If cached reads are ~10% of the normal input price, the prefix cost across the conversation falls by roughly 80–85%. The win scales with prefix size and conversation length.

Model tiering compounds it: route simple turns to Haiku 4.5, default to Sonnet 4.6, and reserve Opus 4.8 for hard turns — the caching discount applies on top of the cheaper tier.

Failure: Redis is a soft dependency — on a cache error, fall through to the model rather than failing the request. Provider prompt-cache misses are invisible (you just pay full price that turn), so never depend on a hit for correctness. A short TTL on the provider side means idle conversations re-prefill on resume; warm them if latency on resume matters.

Likely follow-ups

  • How would you measure cache hit rate, and what hit rate makes prompt caching worth the write premium?
  • The product wants per-user personalization in the system prompt. How do you keep caching while injecting it? (Move it below the breakpoint, or maintain per-user prefixes.)
  • How do you prevent the response cache from returning another tenant’s answer? (Tenant ID in the key; never share across trust boundaries.)
Why does putting personalized text at the top of the prompt hurt, even if the rest of the prompt is huge and stable?

Provider prompt caching keys on an exact prefix of the token stream. The cache can only be reused up to the first byte that differs. If a personalized line sits at the top, every request differs from token one, so nothing downstream — even a 100K-token stable document below it — can be served from cache. Keep all volatile content at the end so the longest possible stable prefix stays cacheable.

2 · A RAG system for enterprise documents

Prompt: Build a system that answers employee questions over our internal docs (wikis, PDFs, tickets) with citations.

Clarifying questions

  • How many documents, how often do they change, and how fresh must answers be (minutes, hours, daily)?
  • Are there access-control boundaries — can every user see every doc, or must retrieval respect per-user permissions?
  • Are citations a hard requirement? (They usually are for enterprise trust.)
  • What does good look like — do we have a labeled eval set, or do we need to build one?

Proposed design — two pipelines

RAG is really two systems: an offline ingest pipeline and an online query pipeline. Design them separately.

  • Ingest: connectors pull source docs → clean & normalize → chunk (semantic or heading-aware, with overlap) → embed each chunk → upsert vectors + metadata (source URL, ACL, timestamp) into a vector index. Re-run incrementally on document changes.
  • Query: embed the question → retrieve top-K by vector similarity (plus optional keyword/BM25 hybrid) → rerank to a tight top-N → assemble a grounded prompt with the chunks and their IDs → generate with citations → return answer + source links.

INGEST (offline)
sources (wiki · PDF · tickets) ──▶ [ connectors ] ──▶ [ clean / normalize ]
  ──▶ [ chunk + overlap ] ──▶ [ embed ] ──▶ [ vector index + metadata/ACL ]

QUERY (online)
user ──▶ [ gateway ] ──▶ [ embed query ] ──▶ [ retrieve top-K · ACL filter ]
  ──▶ [ reranker → top-N ] ──▶ [ prompt assembler ] ──▶ [ LLM ] ──▶ answer + citations

EVAL (continuous)   labeled Q/A set ──▶ retrieval recall@K + answer faithfulness

Three loops: ingest builds the index, query serves answers, eval keeps both honest. Metadata/ACL travels with every chunk so permissions are enforced at retrieval.

Key tradeoffs

  • Chunk size. Small chunks give precise retrieval but fragment context; large chunks preserve context but dilute relevance and waste tokens. Heading-aware chunking with overlap is the usual compromise.
  • Top-K vs. cost/latency. Retrieving more candidates lifts recall but inflates rerank work and prompt tokens. Retrieve wide, rerank to a small N, and stuff only the N best into context.
  • Hybrid vs. pure vector. Pure vector search misses exact identifiers (error codes, SKUs); adding BM25 keyword search recovers them, at the cost of a second index and fusion logic.
  • Freshness vs. ingest cost. Re-embedding everything is expensive; incremental upserts keyed on a content hash keep the index fresh without full rebuilds.

Scaling, failure & cost

Scale: ingest is embarrassingly parallel — fan out embedding across workers and batch upserts. The vector index is the stateful bottleneck; pick one that shards and supports metadata filtering at scale. Cache embeddings of repeated queries.

Failure: if retrieval returns nothing relevant, the model must say I don’t know rather than hallucinate — enforce grounding in the system prompt and verify with eval. Stale index → wrong answers; alert on ingest lag. Citations let users catch errors the system cannot.

Cost & eval: generation tokens dominate, so trim context to the reranked top-N and use prompt caching for the stable instruction block. Use Haiku 4.5 for cheap retrieval-quality checks and Sonnet 4.6 or Opus 4.8 for final generation. Run a standing eval: retrieval recall@K (did the right chunk make the candidate set?) and answer faithfulness (is every claim supported by a cited chunk?), the latter scored by an LLM judge against a labeled set.

Likely follow-ups

  • A user asks about a doc they are not allowed to see. Where do you enforce that? (At retrieval — ACL filter on the index — never just by hoping the model declines.)
  • Retrieval recall is fine but answers are wrong. Where do you look? (Reranking, chunking, prompt grounding, or the generator — bisect with eval.)
  • How do you keep answers fresh when a doc changes? (Incremental re-embed on content-hash change; invalidate the affected chunks.)
Retrieval recall@20 is 95%, but users still get wrong answers. What is the most likely culprit, and how do you confirm?

If the right chunk is in the candidate set 95% of the time, the failure is downstream of retrieval: either the reranker is dropping the right chunk before it reaches the prompt, or the generator is ignoring the provided context. Confirm by measuring two things separately — rerank precision (is the correct chunk in the final top-N?) and answer faithfulness given a known-good context. If faithfulness is low even when the right chunk is present, fix grounding (prompt instructions, model tier); if the chunk vanishes at rerank, fix the reranker or N.

3 · An agent platform

Prompt: Design a platform that lets internal teams build and run LLM agents that call tools, with safety and cost controls.

Clarifying questions

  • Who writes the tools — a trusted internal team, or untrusted tenants? (Decides how hard the sandbox must be.)
  • Are agents interactive (a user waits) or background (long-running jobs)? Affects timeouts and the execution model.
  • Multi-tenant? If so, what isolation do tenants expect for data, secrets, and spend?
  • What are the hard caps — max steps per run, max dollars per run, max wall-clock?

Proposed design

An agent is a loop: the model proposes a tool call, the platform executes it, feeds the result back, and repeats until the model emits a final answer. The platform’s job is to make that loop safe, observable, and bounded.

  • Tool registry: declarative tool definitions (name, JSON schema, auth scope, cost weight). The model only ever sees schemas; the platform owns execution.
  • Sandboxed executor: each tool call runs in an isolated sandbox (container / microVM) with no ambient credentials, an egress allowlist, and a wall-clock timeout. Untrusted tools get a fresh, throwaway environment.
  • Orchestrator: drives the loop, enforces the step budget and the cost cap, handles retries, and writes a trace span for every step.
  • Budget/cost governor: a running tally of tokens and tool spend per run and per tenant; the loop halts when a cap is hit.
  • Tracing: every model call and tool call is a span with inputs, outputs, tokens, latency, and cost — the only way to debug or price an agent.

tenant request ──▶ [ API gateway · authn/z + per-tenant quota ]
  ▼
[ orchestrator (agent loop) ] ◀──────────────┐
  │ proposes tool call           observation
  ├──▶ [ budget/cost governor ]  steps + $ caps
  ├──▶ [ tool registry ]  schema + scope lookup
  └──▶ [ sandboxed executor ] ──▶ external API / DB ──┘
  │
  ▼
[ tracing / observability ] ◀── spans from every model & tool call
  │
  ▼
[ LLM ]  Haiku 4.5 routing · Sonnet 4.6 default · Opus 4.8 hard steps

The orchestrator is the hub; the governor, registry, sandbox, and tracer wrap every step. Tenancy is enforced at the gateway and carried through every span.

Key tradeoffs

  • Isolation strength vs. cold-start latency. A fresh microVM per call is the safest and the slowest; a warm pooled sandbox is fast but leaks state across calls if not reset. Match isolation to trust level.
  • Autonomy vs. control. A high step cap lets agents solve harder tasks but risks runaway loops and cost; a low cap is safe but brittle. Make it configurable per tool/tenant.
  • Model tier vs. cost. Running every step on Opus 4.8 is capable but expensive; route routine steps to Haiku 4.5 and escalate to Opus 4.8 only when a step is hard or a cheaper model failed.
  • Retry vs. duplicate side effects. Retrying a tool that already mutated state can double-charge or double-send — require idempotency keys for any tool with side effects.

Scaling, failure & cost

Scale: the orchestrator is stateless per step — persist run state externally (DB/queue) so any worker can resume a run; this also makes long background agents survivable across deploys. Pool sandboxes by trust tier.

Failure: a stuck or looping agent is the signature failure — the step cap and cost cap are the backstop. Tool timeouts prevent one hung dependency from hanging the run. A poisoned tool result can derail the model, so validate tool outputs against their schema before feeding them back.

Cost: tokens grow with every loop iteration because history accumulates, so cost is roughly per-step tokens × steps — cap steps, summarize or trim history, and cache the stable system/tool-schema prefix. Per-tenant cost caps prevent one team from draining the budget.

Likely follow-ups

  • An agent is stuck calling the same tool forever. How does your design stop it and how do you debug it? (Step/cost cap halts it; the trace shows the loop.)
  • A tenant’s tool tries to read another tenant’s data. How is that prevented? (No ambient creds; per-call scoped credentials; egress allowlist.)
  • How do you bill tenants accurately? (Sum token + tool cost from the per-run trace spans.)
Why feed scoped, per-call credentials into the sandbox instead of giving the executor a broad service account?

A broad service account means any tool — including a buggy or malicious one written by a tenant — inherits the union of all permissions, so a single compromised tool can reach data and systems it has no business touching. Scoping a narrow, short-lived credential to each call enforces least privilege: the blast radius of a misbehaving tool is exactly what that one tool was authorized to do, and nothing more. Combined with an egress allowlist and no ambient credentials in the sandbox, it contains both accidents and attacks.

4 · A multi-tenant rate limiter

Prompt: Design a rate limiter for our LLM API that is fair across tenants and protects the backend.

Clarifying questions

  • Limit per API key, per user, or per org? (Almost always per tenant, sometimes nested.)
  • Limit on requests per minute, tokens per minute, or both? For LLMs the real scarce resource is tokens, not requests.
  • Single region or global? (Decides whether the counter must be distributed and consistent.)
  • On exceed: reject with 429, or queue/throttle? What does the client contract promise?

Proposed design

For LLM APIs you need two limits enforced together: requests-per-minute (RPM) and tokens-per-minute (TPM). A request passes only if both buckets allow it.

  • Algorithm: token-bucket per tenant for each dimension. It allows bursts up to the bucket size and refills at a steady rate — a better fit for bursty LLM traffic than a hard fixed window, and cheaper than a precise sliding-window log.
  • Token cost is estimated up front, reconciled after. Charge the TPM bucket the estimated input + max output tokens at admission; refund the difference once the real output token count is known. This stops a few huge requests from blowing the budget.
  • Distributed counter: hold buckets in Redis; do the check-and-decrement atomically (a Lua script) so concurrent requests across API servers cannot oversell the limit.
  • Response contract: on exceed, return 429 with Retry-After and the limit headers, so well-behaved clients back off instead of hammering.

clients (tenant A · B · C) ──▶ [ API servers · N replicas ]
  │ each server runs the same limiter check
  ▼
[ rate-limiter middleware ]
  │ atomic check-and-decrement (Lua)
  ▼
[ Redis ]   per-tenant buckets:   {RPM}   {TPM}
  │ both allow? ──no──▶ 429 + Retry-After
  ▼ yes
[ LLM backend ] ──▶ stream ──▶ reconcile TPM (refund unused)

fairness: per-tenant buckets isolate noisy neighbors; optional global ceiling protects the backend

Stateless API replicas share one source of truth in Redis. Per-tenant buckets give fairness; a global ceiling gives backpressure for the whole backend.

Key tradeoffs

  • Token bucket vs. sliding window. Token bucket allows controlled bursts and is cheap (two numbers per tenant: level + last-refill time). A sliding-window log is more precise at the boundary but stores a timestamp per request — more memory and more work. Token bucket is the usual pick; sliding-window-counter is a middle ground if boundary bursts are a problem.
  • Requests vs. tokens. Limiting only RPM lets one tenant send a few enormous prompts and starve the backend; limiting only TPM lets a flood of tiny requests overwhelm connection/queue capacity. You need both.
  • Strict consistency vs. latency. A central Redis check is one extra network hop per request but prevents overselling; local in-memory buckets are faster but drift across replicas and over-admit. For correctness across N servers, centralize.
  • 429 vs. queue. Rejecting is simple and honest; queueing smooths bursts but adds latency and a new failure mode (unbounded queues). Bound any queue and shed load past it.

Scaling, failure & cost

Scale: shard Redis by tenant key so the counter scales horizontally; the check is O(1). Estimate-then-reconcile keeps the hot path to a single atomic op.

Failure: if Redis is unreachable, choose your failure mode deliberately — fail-open (admit, protect availability, risk overload) or fail-closed (reject, protect the backend, hurt availability). A local fallback bucket per server is a reasonable middle: degrade to approximate limiting rather than none. Clock skew across servers can mis-refill buckets; refill based on stored timestamps, not wall-clock deltas computed locally.

Fairness & cost: per-tenant buckets stop a noisy neighbor from consuming a shared pool; add a global ceiling so the sum of tenants cannot exceed backend capacity (backpressure). Surfacing Retry-After turns client retries from a thundering herd into orderly backoff, which is itself a cost and stability win.

Likely follow-ups

  • Why limit tokens, not just requests, for an LLM API? (Tokens are the scarce, variable-cost resource; request count hides 100× cost differences.)
  • Two API servers check the limit at the same instant. How do you avoid overselling? (Atomic check-and-decrement in Redis via a Lua script.)
  • Redis is down. What happens? (A deliberate fail-open or fail-closed choice, ideally with a local degraded fallback.)
Why reserve the estimated token cost at admission and refund later, instead of just charging the actual tokens after the request finishes?

If you only charge after completion, many large requests can all be admitted before any of them finishes — you learn you blew the limit only in hindsight, after the backend is already overwhelmed. Reserving the estimate (input + max output) at admission means the budget is debited before work starts, so concurrent big requests are correctly rejected once the bucket is empty. Refunding the unused portion afterward keeps tenants from being overcharged for reserving more than they used. Reserve-then-reconcile gives you both safety (never oversell) and fairness (never overcharge).

How this shows up in the interview

  • They will interrupt with a constraint mid-design. (Now it must be multi-region.) That is a test of whether your design has clean seams. Name what changes (the distributed counter, cache locality) and what does not.
  • They will ask you to estimate cost out loud. Have the token-math reflex: input + output tokens × price × volume, then the biggest lever. Knowing prompt caching cuts repeated-prefix cost dramatically is a strong, specific signal.
  • They will probe a failure mode you skipped. Pre-empt it: for every external call, say the timeout, the retry policy, and the fallback before they ask.
  • They reward comparison, not assertion. Say token bucket vs. sliding window, fail-open vs. fail-closed, Haiku for routing vs. Opus for hard steps — and then commit to one with a reason.
