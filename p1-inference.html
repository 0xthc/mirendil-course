Inference, latency & cost

When a product feels slow or a bill feels scary, the cause is almost always one of a few mechanical facts about how a model turns a prompt into tokens. Learn these and you can diagnose latency, predict cost, and pick the right model on sight — the single clearest signal that you actually understand the thing you are building on.

ELI5

Asking a model a question is like asking someone to read your whole letter and then handwrite a reply. Reading the letter is fast — they can skim the whole page at once. Writing the reply is slow — one word at a time, and every new word means re-reading everything written so far. The reading phase is prefill; the writing phase is decode. Almost all the waiting is the writing, and you pay the most for the words the model writes back.

The two phases: prefill and decode

Every request runs in two distinct phases with completely different performance characters. Confusing them is the root of most wrong intuitions about speed and cost.

PREFILL (read the prompt)DECODE (write the answer)
What happensThe whole prompt is processed in one shot.Output tokens are generated one at a time, each depending on the last.
ParallelismAll prompt tokens at once — highly parallel.Strictly sequential — token N needs token N-1.
BottleneckCompute-bound (the GPUs are busy doing math).Memory-bandwidth-bound (the GPUs wait on reading weights + cache).
You feel it asTime to first token (TTFT).Tokens per second, and total length of the wait.

Here is the part people miss. Prefill is parallel, so a 2,000-token prompt is processed in roughly the same wall-clock time as a 200-token one would need passes — it is one big matrix multiply that keeps the GPU saturated. Decode cannot be parallelized over time: to write the 100th word the model must have already written the 99th. So each output token is its own forward pass through the entire model, and those passes happen back-to-back. That is why decode dominates latency — a 500-token answer is 500 sequential trips through the network.

The two big levers

Two facts drive nearly every latency-and-cost decision you will make: (1) output tokens are far more expensive and far slower than input tokens because each one is a separate sequential forward pass, and (2) prompt caching lets you skip re-paying for a stable prefix you have already processed. Shorten outputs and cache prefixes, and you have pulled the two largest levers that exist.

Why decode is memory-bound (and prefill is not)

During decode the model generates one token, which is a tiny amount of arithmetic — but to do it the GPU must read every weight in the model out of memory, plus the cache of all previous tokens. The math finishes long before the reading does, so the chip sits idle waiting on memory bandwidth. Prefill, by contrast, reads those same weights once and reuses them across thousands of prompt tokens in parallel, so the math is the bottleneck instead. This is also why batching helps decode so much: if you read the weights once and apply them to many requests at the same time, the expensive memory read is amortized.

The KV cache

To predict the next token, the model's attention mechanism needs the keys and values of every previous token. Recomputing those from scratch on every step would make decode quadratically slow. So the model stores them: after a token is processed, its keys and values are kept in the KV cache, and each new token only computes its own K/V and attends against the saved ones. That is what makes per-token decode cheap and roughly constant instead of growing with position.

Token 1 → compute K,V → store in cache
Token 2 → compute its K,V → store → attend over [1,2]
Token 3 → compute its K,V → store → attend over [1,2,3]
… each step adds one row; nothing is recomputed.

The KV cache trades memory for speed: keys/values for past tokens are saved, so each new token is cheap.

The catch: the cache grows linearly with context length. A long conversation or a big document means a big cache, held in GPU memory for the whole request. That cache is what bounds how many requests can run at once — it competes with other requests for the same finite memory. So context length does not just cost you input tokens; it eats the memory budget that determines throughput. Longer context → bigger cache → fewer concurrent requests → lower throughput and, often, higher latency under load.

Why output tokens cost about 5× input

Look at any frontier model's pricing and output tokens are several times more expensive than input tokens — commonly around 5×. The lifecycle above explains exactly why. Input tokens are consumed in the parallel, compute-efficient prefill: read once, processed together. Output tokens each require a full sequential forward pass over the whole model, and that pass is memory-bandwidth-bound and cannot be batched across time. You are paying for scarce sequential GPU time, not parallel throughput. The same asymmetry means output tokens dominate latency too: a request that reads 4,000 tokens and writes 50 often feels snappy, while one that reads 200 and writes 2,000 feels slow.

Why long prompts are expensive — and how prompt caching helps

A long prompt is expensive in two ways: you pay input-token cost for every token of it on every call, and prefilling it takes real time that lands on TTFT. If your system prompt, tool definitions, and a big chunk of context are the same on every request — which is the common case for an agent or a chat product — you are re-paying to prefill identical text again and again.

Prompt caching fixes this. The provider caches the prefill state (the computed KV) of a stable prefix. On the next call with the same prefix, that work is reused instead of recomputed: cached input tokens are dramatically cheaper (often around a tenth of the price) and TTFT drops because most of the prefill is already done. The win is largest exactly where it hurts most — long, stable system prompts and long-context documents reused across many turns.

Gotcha

Prompt caching only helps a prefix that is byte-identical and stable. If you put something that changes every request — a timestamp, the user's name, a freshly retrieved snippet — near the top of your prompt, you invalidate the cache for everything after it and pay full prefill again. Order your prompt cache-friendly: stable content first (system prompt, tools, fixed context), volatile content last. Caches also expire after a short idle window, so a low-traffic feature may rarely get a hit.

Throughput vs latency: they are not the same goal

Latency is how fast one request finishes. Throughput is how many tokens the system delivers in total per second across all users. They trade off, and the main dial is batching: running many requests together amortizes the expensive weight reads in decode, raising throughput — but a request may wait to join a batch, and a fuller batch can slow each individual decode step, raising latency. A serving system tuned for cost-per-token (big batches) and one tuned for a snappy single user (small batches, low queueing) are configured differently.

Two latency numbers matter, and you should name both:

  • Time to first token (TTFT) — responsiveness; how long until the user sees something. Driven by queue wait + prefill (so by prompt length and caching).
  • Tokens per second (throughput per stream) — how fast the answer flows once it starts. Driven by model size, batch pressure, and decode efficiency.

And always report percentiles, not averages. A p50 (median) of 400 ms with a p99 of 6 s means one user in a hundred is having a terrible time — usually from queueing under load or unusually long outputs. Averages hide exactly the tail that makes users churn.

Choosing a model: the cost / latency / quality triangle

You rarely need the biggest model for every call. Match the tier to the job. Using Claude as the running example across three tiers:

Small / fastHaiku 4.5Classification, routing, extraction, simple tool calls, high-volume tasks, sub-tasks inside a larger pipeline.Cheapest, lowest latency.
Mid / balancedSonnet 4.6The everyday workhorse: most chat, drafting, summarization, solid reasoning and coding at good speed.Moderate.

The senior move is to route by difficulty: send the easy 80% of traffic to a small model and escalate only the hard cases to a large one. A classifier that decides which tier to use can cut cost and latency dramatically while keeping quality where it counts. Pair that with caching and tight output limits and you have the three biggest knobs working together.

How this shows up in the interview

This topic is a favorite because it instantly separates people who have shipped on models from people who have only called an API. Expect questions like these.

Q: Why is long context slow and expensive?

Two mechanisms. Cost: every input token is billed and must be prefilled; long prompts mean more prefill work and a larger KV cache held for the whole request. Latency: a long prompt lengthens prefill, which is most of TTFT, and the large KV cache consumes GPU memory that would otherwise allow more batching — so throughput drops and tail latency rises under load. The fix toolbox: prompt caching for stable prefixes, retrieval to send only relevant context instead of everything, and trimming history.

Q: Users say the agent feels slow. How do you diagnose it?

Separate the two clocks first. Is TTFT high or is tokens/sec low? If TTFT is high, the prompt is too long or uncached, or requests are queueing — shorten/cache the prefix, check load. If tokens/sec is low or the answer is just long, the output is the problem — cap max_tokens, instruct for brevity, or drop to a smaller/faster model. Then check the agent shape: are tool calls serial when they could run in parallel? Is every call hitting the largest model when a small one would do? Always look at p99, not the average, and confirm streaming is on so the user sees tokens immediately. Concretely: enable prompt caching, shorten outputs, parallelize tool calls, route easy steps to a smaller model.

Q: Pick a model for feature X.

State the tradeoff out loud: cost vs latency vs quality, driven by volume and how costly a wrong answer is. A high-volume classifier or router → small/fast tier (Haiku 4.5). A user-facing chat or drafting assistant → balanced tier (Sonnet 4.6). A complex coding agent or a high-stakes reasoning step → most-capable tier (Opus 4.8). Then add the kicker: start on a mid tier, measure quality with evals, and only move up if quality demands it or down if cost does — and consider routing by difficulty rather than picking one tier for everything.

Check your understanding

A request reads a 4,000-token document and writes a 40-token summary. Another reads a 100-token question and writes a 1,500-token essay. Which is slower and more expensive, and why?

The essay request — by a lot. Output tokens are generated one at a time, each a separate sequential forward pass, so 1,500 output tokens means 1,500 sequential trips through the model; that dominates both latency and (at ~5× the price) cost. The document's 4,000 input tokens are processed in one parallel, compute-efficient prefill and are billed at the cheaper input rate. Reading is cheap and fast; writing is expensive and slow.

What is the KV cache, and why does a longer context reduce how many requests you can serve at once?

The KV cache stores the keys and values of all previously processed tokens so each new decode step only computes its own K/V and attends over the saved ones — that is what makes per-token decode cheap. But the cache grows linearly with context length and lives in GPU memory for the whole request. More context per request means a bigger cache per request, leaving less memory to hold other requests' caches — so fewer requests fit in a batch, throughput falls, and tail latency rises under load.

You add prompt caching to your agent but see almost no improvement. What likely went wrong?

The cacheable prefix probably is not actually stable. If something volatile — a timestamp, the user's name, a freshly retrieved snippet — sits near the top of the prompt, it invalidates the cache for everything after it, so each call re-prefills from there. Reorder so stable content (system prompt, tool definitions, fixed context) comes first and volatile content last. Also check traffic: caches expire after a short idle window, so a low-traffic feature may rarely get a cache hit.
