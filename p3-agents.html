Agents

A single model call answers a question. An agent chases a goal: it calls a tool, looks at what came back, decides what to do next, and keeps going until the job is done. That loop is most of the value — and most of the danger. This page is about building the loop so it finishes, stays cheap, and doesn't melt down when a tool misbehaves.

ELI5

A plain model call is like asking a smart friend a question and getting one answer. An agent is like handing that friend a phone, a calculator, and your credit card and saying "book me a table somewhere good tonight." Now they can look things up, try things, and react to what they find — but you also really want a spending limit and a "stop after 10 phone calls" rule, because they will happily keep dialing forever if something goes wrong.

One model call vs. the agent loop

A single call is stateless and one-shot: prompt in, text out. It can't check anything in the world, can't react to a result, and can't recover if it guessed wrong. An agent wraps the model in a loop that gives it hands and a feedback channel:

# a single call — no feedback, no actions
answer = model(prompt)

# an agent — observe, think, act, repeat
messages = [system_prompt, user_goal]
for step in range(MAX_STEPS):        # loop guard (see below)
    reply = model(messages, tools=registry)
    if reply.is_final:                # model says it's done
        return reply.text
    result = run_tool(reply.tool_call)   # ACT in the world
    messages.append(reply)               # remember what it asked
    messages.append(result)              # OBSERVE the result, feed it back
raise StepLimitExceeded            # never trust the loop to end on its own

The four beats of the loop: observe (read the latest tool result), think (the model reasons over the running transcript), act (it emits a tool call), get result (you run the tool and append the output). Then repeat. The whole conversation — every call, every result — accumulates in messages, which is why agents get expensive: each step re-sends the entire growing transcript.

The one idea to keep

An agent is a loop where the model chooses the next action and the world answers back. The model never touches your systems directly — it can only request a tool call as structured data. Your code decides whether to run it, runs it, and hands back the result. That gap between "the model asked" and "the system did" is where every safety control — validation, sandboxing, approvals, retries, budgets — lives.

The tool registry: what the model is allowed to do

The model can only call tools you have declared. A tool declaration is a name, a description, and a JSON Schema for its arguments. The description and schema are the entire interface — the model picks a tool by reading them, so they are prompt-engineering, not afterthoughts.

def get_orders(customer_id: int, limit: int = 10): ...

# what the model actually sees:
{
  "name": "get_orders",
  "description": "List a customer's orders, newest first. Use after you have a customer_id.",
  "input_schema": {
    "type": "object",
    "properties": {
      "customer_id": {"type": "integer"},
      "limit": {"type": "integer", "default": 10}
    },
    "required": ["customer_id"]
  }
}

The model returns a structured call — get_orders(customer_id=4471) — that your runtime parses and dispatches. Practical registry rules:

  • Few, sharp tools beat many fuzzy ones. A model staring at 60 overlapping tools picks wrong; 8 well-named ones with clear descriptions go far. If two tools overlap, the model will dither between them.
  • The description is the contract. Say when to use it, what it returns, and its preconditions ("use after you have a customer_id"). Schema constraints (enums, required fields, ranges) prevent malformed calls before they ever run.
  • Return structured, compact results. Every tool result is re-sent on the next turn, so a tool that dumps 20 KB of JSON inflates the cost of every later step. Return the fields the model needs, not the raw payload.

Termination & loop guards: making sure it stops

The model decides when it's done — and sometimes it never decides. A flaky tool, an impossible goal, or a confused plan can put the agent into a loop that calls the same tool forever. You are responsible for stopping it. Layer several independent guards; do not rely on the model to behave:

  • Max steps. A hard cap on loop iterations (e.g. 10–25). Simple, and it catches everything eventually.
  • No-progress / repeat detection. If the model issues the same tool call with the same arguments twice, or cycles A→B→A→B, it's stuck. Detect the repeat and break — feeding back "you already tried that" sometimes unsticks it, otherwise bail.
  • Budget caps. Track cumulative tokens and dollars across the whole run; stop when you cross a ceiling (stop after $0.50 or 100K tokens). Time budgets too — a wall-clock deadline for the whole task.
Gotcha: the infinite loop that bills you

The classic agent disaster: a tool starts returning an error the model can't fix, so the model retries it, gets the same error, retries again… forever. With no step cap and no budget cap, a single stuck request can fire thousands of model calls overnight and run up a real bill — each call re-sending an ever-growing transcript, so it gets more expensive per step, not less. Always set MAX_STEPS and a dollar ceiling before you ship, not after the invoice. Treat hitting a guard as a normal, expected exit — log it, return a partial answer, move on.

Cost and latency grow every step

Because each model turn re-sends the full transcript, an agent's cost per step climbs as the conversation grows — and total latency is the sum of every model turn plus every tool's runtime. A 6-step agent is not 6× one call; it's more, because the later calls carry more context. This is the single most surprising thing for engineers new to agents.

Levers that actually move the bill: prompt caching (the system prompt + tool defs + early transcript are stable, so cache them and pay a fraction on every later step); compact tool results (don't re-send 20 KB you'll re-send again); a smaller model for the loop (Haiku 4.5 for routine tool-picking, escalate to Sonnet/Opus only when a step needs deeper reasoning); and fewer steps via parallel tool calls and good planning.

When tools fail: retries, fallbacks, surfacing

Real tools error, time out, rate-limit, and return junk. The agent's reliability is mostly your error-handling reliability. A tiered response:

  • Retry transient failures — timeouts, 503s, rate limits — with exponential backoff and a small retry cap. Don't retry a 400 (bad arguments): that's deterministic, it'll fail identically.
  • Feed real errors back to the model as a tool result. Often the best recovery is to return {error: "no customer named Acme; try a fuzzy search"} and let the model adapt — it'll pick a different tool or fix its arguments. The model is a recovery mechanism, not just a caller.
  • Fall back to a degraded path (cached data, a simpler tool, a secondary provider) when the primary is down.
  • Surface to the user when you're out of options. A wrong-but-confident answer is worse than "I couldn't reach the orders system — try again shortly." Never let a tool failure get silently laundered into a hallucinated result.
Validate tool output, don't just trust it

A tool returning HTTP 200 doesn't mean the result is sane. An empty list, a malformed row, or a stale value will get fed straight back to the model, which may build a confident answer on garbage. Validate and shape tool results before they re-enter the transcript — and remember that tool output (a web page, a doc, a DB field) can carry injected instructions. Treat it as untrusted data, never as commands to obey.

Parallel vs. sequential tool calls

If the next actions don't depend on each other — check weather in three cities, fetch a user's orders and their support tickets — the model can request several tool calls in one turn and your runtime can run them concurrently. Latency drops from the sum to the max:

Sequential: call A (300 ms) → call B (300 ms) → call C (300 ms) = ~900 ms

Parallel: A, B, C dispatched together, await all = ~300 ms

Only safe when the calls are independent. If B needs A's result, they're a chain and must stay sequential.

The catch: parallelism helps only for independent work. Most multi-step agent tasks are a dependency chain (you need the customer_id before you can fetch their orders), and those stay sequential no matter what. Look for fan-out — one step that spawns several independent lookups — and parallelize there.

Planning vs. reactive

Two postures, and good agents mix them:

  • Reactive (the default loop above): decide one action, see the result, decide the next. Robust to surprises because every decision uses fresh information — but it can wander on a long task with no map.
  • Planning: ask the model to lay out the steps first ("1. find customer, 2. pull orders, 3. sum totals"), then execute against that plan, re-planning when reality diverges. This shines on multi-step tasks with clear structure — it keeps the agent on-track and makes its intent inspectable.

Rule of thumb: plan when the task is long and structured; stay reactive when it's short or unpredictable. A plan that the agent can't deviate from is brittle; the strong pattern is "plan, then react against the plan, re-planning when a step fails."

Observability: you cannot operate what you cannot see

An agent is a non-deterministic, multi-step, multi-cost process. When one misbehaves in production — wrong answer, $40 run, 90-second latency — you need to replay exactly what happened. Trace every step: the model's input and output, each tool call's arguments and result, per-step tokens, cost, and latency, and which guard (if any) ended the run.

Why tracing is non-negotiable in production

Without traces, every agent bug is unreproducible — the run is gone and the next run takes a different path. With traces you can answer the only questions that matter on-call: where did it go wrong, how much did it cost, and how long did it take? A trace also turns vague "the agent is dumb" complaints into a concrete failing step you can fix, and it's the raw material for evals — you replay real traces to measure whether a prompt or tool change made the agent better or worse.

Human-in-the-loop: approvals for risky actions

Some actions are irreversible: sending money, deleting records, emailing a customer, deploying code. For those, the agent shouldn't act autonomously — it should pause and ask. Classify tools by blast radius: read-only tools run freely; mutating or external-facing tools require an approval gate where a human sees the exact action and its arguments and clicks yes/no before your runtime executes it.

This is the same model/runtime gap from the start of the page: the model can only request the action; your code holds it at the gate. Approvals are cheap insurance against the one runaway action that can't be undone — and they pair naturally with sandboxing (run tool code with least privilege) and per-tool budgets.

How this shows up in the interview

Q: An agent in production sometimes loops forever or gives wrong answers when a tool fails. How do you make it reliable?

Attack it in layers. Loops: a hard MAX_STEPS cap plus repeat-detection (same tool + same args twice ⇒ break) plus a wall-clock and dollar budget — independent guards so one catches what another misses. Tool failures: retry transient errors (timeout, 503, rate-limit) with backoff and a retry cap; don't retry deterministic 400s; feed real errors back to the model as a tool result so it can adapt; fall back to a degraded path; and surface honestly to the user when out of options rather than letting a failure become a hallucinated answer. Validate tool output before it re-enters the transcript so garbage doesn't drive a confident wrong answer. And make all of this visible with tracing so you can actually diagnose the next failure. The meta-point interviewers want: the model is unreliable by nature; reliability is engineered in the loop around it.

Q: This agent is too expensive. How do you cap and reduce cost?

First cap it so it can't surprise you: track cumulative tokens and dollars per run and stop at a ceiling; cap steps. Then reduce: enable prompt caching on the stable prefix (system prompt + tool defs + early transcript) since every step re-sends it — this is the biggest single win because cost-per-step otherwise climbs as the transcript grows; keep tool results compact so you're not re-sending the same KBs each turn; use a cheaper model (Haiku 4.5) for routine tool-selection and escalate to a stronger one only for hard steps; and cut the number of steps via parallel tool calls and lightweight planning. Show the cost growth with a per-step breakdown and you've demonstrated you understand why agents are expensive, not just that they are.

Q: Design an agent platform other teams build on. What are the core pieces?

A loop runtime plus the guardrails teams shouldn't have to reinvent: a tool registry with schema validation and per-tool permission/blast-radius tags; tool sandboxing (least-privilege execution, network/filesystem isolation) so a buggy or malicious tool can't reach beyond its mandate; built-in retries/timeouts/fallbacks with sane defaults; loop guards (max steps, repeat detection) and budgets (tokens, dollars, wall-clock) enforced centrally per run and per tenant; an approval/human-in-the-loop gate for tools flagged risky; and tracing/observability for every step (inputs, tool calls, cost, latency, exit reason) feeding dashboards, alerts, and an eval harness. The framing to lead with: the platform's job is to make the safe path the default — teams declare tools and goals; the platform owns termination, cost control, isolation, and visibility.

Check your understanding

Why does cost per step tend to rise as an agent runs, and what's the single most effective lever against it?

Each model turn re-sends the entire growing transcript (every prior call and tool result), so later turns carry more input tokens and cost more than earlier ones — total cost curves upward, not linearly. The most effective lever is prompt caching: the system prompt, tool definitions, and early transcript are a stable prefix, so caching them means you pay only a fraction to re-send them each step. Compact tool results and using a cheaper model for routine steps help too, but caching attacks the growth directly.

A tool keeps returning an error the model can't resolve. Name two independent guards that would stop the agent, and one thing you should NOT rely on.

Independent guards: (1) a max-steps cap that breaks the loop after N iterations regardless of state, and (2) a budget cap (cumulative tokens/dollars/wall-clock) that stops the run when it crosses a ceiling. Repeat-detection (same tool + same args twice ⇒ break) is a great third. What you should NOT rely on: the model deciding to stop on its own — a stuck model will happily retry forever, which is exactly the runaway-cost failure mode.

When is it safe to run tool calls in parallel, and why are most multi-step agent tasks still sequential?

Parallel is safe only when the calls are independent — none needs another's result (e.g. weather in three cities). Then latency drops from the sum of the calls to the max. Most agent tasks are dependency chains: you need the customer_id before you can fetch their orders, so those steps must stay sequential. The win is to spot fan-out — a single step that spawns several independent lookups — and parallelize just that.

The model "deleted a production record." Where in the agent architecture should that have been prevented, and how?

At the model/runtime gap: the model can only request a tool call as structured data — your code decides whether to run it. A destructive tool like delete_record should be tagged risky and sit behind a human-in-the-loop approval gate (a human sees the exact action + arguments and confirms before execution), ideally combined with sandboxing/least-privilege so the tool can't exceed its mandate. Reversible/read-only tools run freely; irreversible ones pause for a human. The model is never trusted to self-restrain on dangerous actions.
