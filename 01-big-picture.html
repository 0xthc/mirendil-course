Module 01

The big picture

Before any code: why does anyone split a neural network across multiple GPUs, and what are the three strategies the test is comparing?

The problem in one sentence

A modern transformer, while it runs, needs more memory than fits on a single GPU. So we cut the work into pieces, put each piece on a different GPU, and have the GPUs talk to each other to stitch the answer back together.

Analogy

Think of rendering a huge video. One machine can't hold the whole project in RAM, so you split it across a render farm. Each box renders some frames, then you collect the results. The hard part isn't the rendering — it's deciding how to split and when the boxes have to exchange data. That coordination is the entire subject of this course.

Two completely different things eat your GPU memory

This distinction is the key that unlocks everything else. When a transformer runs, GPU memory is consumed by two unrelated categories:

CategoryWhat it isWhat makes it grow
Model stateThe weights (and during training, their gradients and optimizer state). These are the learned parameters — the matrices W_q, W_k, etc.Bigger model (more layers, wider hidden size). Does not depend on how long your input is.
ActivationsThe intermediate tensors produced while processing a specific input — the Q/K/V tensors, the attention scores, etc. Scratch data, not learned.Longer input sequence (more tokens) and bigger batch. Does not depend on having more weights.

Here is why this split matters so much. Suppose you want to serve a model with a very long context — the report mentions models targeting 12 million tokens of context. The weights didn't change. But the activations explode, because attention has to deal with that many tokens at once. You can run out of memory from activations alone, even if the weights fit comfortably.

The core insight

Different parallelism strategies attack different memory categories. If you only know one strategy, you can only relieve one kind of pressure. The whole point of the paper is a strategy that relieves both at once.

The three strategies (a first, intuitive pass)

You'll build each of these properly in later modules. For now, just get the shape of the idea.

Tensor Parallelism (TP) — split the weights

Cut each weight matrix into slices and give one slice to each GPU. Now no GPU holds a full weight matrix, so model-state memory shrinks. But every GPU still sees the whole input sequence, so activation memory is not reduced. Attention is computed by splitting it across attention heads — GPU 0 owns some heads, GPU 1 owns others.

Sequence Parallelism (SP) — split the tokens

Cut the input sequence into chunks of tokens and give one chunk to each GPU. Now no GPU holds the whole sequence's activations, so activation memory shrinks. But every GPU still keeps a full copy of all the weights, so model-state memory is not reduced. The catch: attention needs every token to "look at" every earlier token, so the GPUs must exchange their token chunks during attention.

Tensor-Sequence Parallelism (TSP) — split both, on the same GPUs

The paper's idea: fold both splits onto the same set of GPUs. Each GPU owns a slice of the weights and a chunk of the tokens. The aim is to shrink both model-state and activation memory at once, using the same 8 GPUs — instead of having to dedicate some GPUs to one job and some to the other.

What the test actually measures

The experiment (from report.md) runs all three strategies on the same hardware and asks: does TSP really deliver lower peak memory, and is that worth the extra communication?

  • Hardware: 8× NVIDIA H100 GPUs.
  • Knobs swept: sequence length (8K → 64K tokens), number of attention heads (16, 32).
  • Measured: peak memory per GPU, and latency (time to run the attention region).
  • Validated: each parallel implementation is checked against a plain single-GPU reference to prove it computes the same answer.

A genuine spoiler from the results, so you know where this is heading: TSP does use the least peak memory, SP uses the most (because of how K/V gather was implemented), and TSP pays for its memory win with extra communication time. Module 8 reads the real numbers.

Check your understanding

A model's weights fit on one GPU, but you keep running out of memory at long context lengths. Which memory category is the culprit, and which strategy targets it?

Activations. Weights are fixed regardless of context length; it's the per-token intermediate tensors (especially in attention) that blow up as the sequence grows. Sequence Parallelism (SP) directly targets activation memory by sharding the token axis. (TSP would help too, since it shrinks both categories.)

Why isn't TP enough on its own for very long sequences?

TP only shrinks model-state memory. Under TP, every GPU still holds the activations for the entire sequence. So as the sequence gets longer, every GPU's activation memory grows just as fast as it would on a single GPU — TP does nothing to stop that.

In plain English, what is TSP trying to achieve that neither TP nor SP can alone?

Shrinking both memory categories at once, on the same set of GPUs. TP shrinks weights but not activations; SP shrinks activations but not weights. TSP folds both reductions together so a single 8-GPU group gets both savings — at the cost of more data movement, which is the tradeoff module 8 examines.
