Track 2 · Building with models · P7

Product system design

A raw model endpoint is not a product. Between your customers and the model sits a platform: it authenticates callers, keeps tenants apart, rate-limits them, caches what it can, routes to the right model, meters every token, and stays up when a provider hiccups. This page teaches those building blocks one at a time, so that when an interviewer says “design the LLM API platform,” you already have the boxes and the lines.

ELI5

Think of the model as a brilliant but expensive chef. You don’t let every customer walk straight into the kitchen. You put a host out front (the gateway): they check your reservation (auth), make sure one table can’t order the entire menu and starve everyone else (rate limits), reuse a sauce that’s already simmering instead of starting over (caching), send simple orders to the line cook and only the hard ones to the head chef (routing), and hand you an itemised bill on the way out (metering). The platform is everything that isn’t the cooking.

The shape of an LLM API platform

Almost every model-backed product converges on the same spine: a client sends a request, a gateway does the cross-cutting work (auth, limits, routing, logging), one or more model providers do the inference, and tokens stream back through the gateway to the client. Everything else on this page hangs off that spine.

The gateway is the one place every request passes through — which is exactly why it owns auth, limits, caching, routing, and metering. Keep it stateless so you can run many copies behind a load balancer; push shared state (counters, cache, usage) into fast stores beside it.
The load-bearing idea

The gateway should be stateless; the state lives in shared stores beside it — rate-limit counters and the prompt cache in a fast in-memory store (e.g. Redis), usage events in a durable log/warehouse. That split is what lets you scale the gateway horizontally and survive the loss of any single instance. Almost every design answer on this page comes back to “stateless gateway, shared fast state.”

Auth & multi-tenancy

A multi-tenant platform serves many customers from shared infrastructure while making each one feel alone. Three layers do the work:

  • API keys. Each tenant gets one or more keys. Store only a hash of the key (treat it like a password), prefix it so it’s greppable in logs without leaking the secret, and support rotation and per-key scopes (e.g. a read-only key, a key limited to one model).
  • Tenant resolution. The first thing the gateway does after auth is turn a key into a tenant_id. Everything downstream — limits, cache namespaces, usage rows, logs — is keyed by that ID. A request with no tenant is a bug, not a fallback.
  • Isolation. Tenants must not see each other’s data, and one tenant’s traffic must not degrade another’s. Namespace the cache per tenant (so prompts never collide or leak across customers), tag every log and trace with the tenant, and enforce per-tenant quotas so a noisy neighbour can’t consume the whole pool.
Gotcha: the shared prompt cache that leaks

A naïve cache key is just hash(prompt). Now two different tenants who happen to send the same system prompt share a cache entry — and depending on your design, one could observe that the other is also using it (a timing side-channel), or worse, you key responses by prompt and serve tenant A’s cached completion to tenant B. Always include the tenant in the cache key (hash(tenant_id + prefix)) unless you have a deliberate, audited reason to share, and never cache the response across tenants by default.

Rate limiting & quotas

Rate limiting protects the platform (and the upstream provider) from being overwhelmed, and enforces fairness so one tenant can’t starve the rest. Two design choices matter: the algorithm and the unit.

Token bucket vs sliding window

AlgorithmHow it worksFeel
Token bucketEach tenant has a bucket that refills at a steady rate (e.g. 100 units/sec) up to a max capacity. Each request spends units. Empty bucket → reject.Allows bursts up to the bucket size, then settles to the refill rate. Cheap: one counter + timestamp per tenant.
Sliding windowCount events in the trailing window (e.g. last 60s). If the count exceeds the limit, reject. A “sliding log” keeps timestamps; a “sliding counter” approximates with weighted buckets.Smoother, harder limit on “requests per minute”; no big burst allowance. Slightly more state/compute.

In practice many platforms use a token bucket for its burst tolerance and cheap state, and reserve sliding windows for hard contractual ceilings. The key is that the counter lives in a shared store so every stateless gateway instance sees the same tally.

Limit by requests AND by tokens

This is the LLM-specific twist. A classic API limits requests per minute (RPM). But LLM cost and load scale with tokens, not requests: one request with a 200k-token context can cost more than a thousand tiny ones. So you enforce two ceilings at once — RPM and TPM (tokens per minute) — and often a separate budget for input vs output tokens, since output is generated serially and is the latency bottleneck.

The wrinkle: you don’t know the exact output token count until the response finishes. So you estimate the cost up front (input tokens are known; output is bounded by max_tokens), reserve that budget at admission, then reconcile against the real usage when the response completes — refunding or charging the difference back to the bucket.

Fairness & backpressure

Per-tenant buckets give you fairness for free: each tenant draws from its own budget, so a spike from one can’t drain another’s. When a tenant is over its limit, return 429 Too Many Requests with a Retry-After header — this is backpressure, an explicit signal that tells well-behaved clients to slow down rather than hammer you harder. Combine per-tenant limits with a global ceiling that protects the upstream provider from the sum of all tenants.

Gotcha: limiting only by request count

If you cap RPM but not TPM, a single tenant sending huge-context requests can blow your provider budget and tank latency for everyone while technically staying “under the limit.” Token-aware limiting is not optional for an LLM platform — it’s the whole point.

Usage metering & billing

Every request consumes input and output tokens, and tokens cost money — at different rates per model and (with caching) per cache state. Metering is how you turn raw traffic into quotas, dashboards, and invoices.

  • Count per request. When a response completes, record input_tokens, output_tokens, and any cache-read / cache-write token counts, plus the model used. The provider returns these — don’t re-tokenize and guess.
  • Emit a usage event. Write one immutable event per request to a durable stream: {tenant_id, api_key_id, model, input_tok, output_tok, cache_read_tok, ts}. This is the source of truth; treat it like an append-only ledger.
  • Aggregate. Roll events up by tenant, by day, by model, by feature. Aggregation feeds both the customer-facing dashboard and the monthly invoice.
  • Attribute cost. Cost = Σ(tokens × per-token price for that model and cache state). Attribution lets you answer “which tenant / which feature is driving spend?” — the question finance and product will both ask.
Meter on the provider’s counts, not your estimate

Your admission-time token estimate is for reserving rate-limit budget. Your bill must use the authoritative counts the provider returns on completion, because the model may stop early (hit a stop sequence), get cut off, or use cached tokens at a discount. Billing the estimate over- or under-charges customers and erodes trust.

Prompt caching layer

Most prompts have a large stable prefix — a long system prompt, a tool schema, a fixed knowledge document — followed by a small dynamic tail (the user’s latest turn). Re-processing the identical prefix on every call wastes time and money. Prompt caching lets the provider reuse the computed state (the KV cache) for a prefix it has seen before, so it only does fresh work on the new tail.

ELI5

Imagine reading a 50-page briefing aloud before answering each question. Prompt caching means: read the briefing once, keep it open on the desk, and for every later question just read the new question and answer. You skip 50 pages of re-reading each time. (This ties straight back to the inference page: the “briefing” is the KV cache for the prefix.)

  • Cache key. Hash the stable prefix together with the tenant and the model (the cached state is model-specific). Mark the prefix/dynamic boundary explicitly so you cache the longest shared span.
  • What to cache. Long, reused, identical-across-calls content: system prompts, tool definitions, few-shot examples, a shared document. Don’t bother caching short or highly variable text — the bookkeeping outweighs the win.
  • Invalidation. Caches are time-bounded (entries expire after an idle window) and content-addressed (change one token in the prefix and the hash changes, so it’s naturally a miss). Bump a version in the key when the system prompt changes so stale prefixes fall out cleanly.
  • The win. A cache hit on the prefix means far fewer tokens are processed at full price and time-to-first-token drops sharply, because the expensive prefill over the prefix is skipped.
Gotcha: putting volatile data in the cached prefix

If you stuff a timestamp, a request ID, or per-user data at the top of the prompt, the prefix changes every call and you never get a cache hit. Order matters: put the truly stable content first (system prompt, tools, shared docs), and the volatile, per-request content last. Cache the head, vary the tail.

Model routing & fallback

Not every request needs your most powerful model. Routing sends each request to the cheapest model that can do the job; fallback keeps the platform up when a model or provider fails.

  • Route by task. Classification, extraction, short replies → a fast, cheap model (e.g. Haiku-tier). Hard reasoning, long synthesis → a strong model (e.g. Opus-tier). The routing decision can be static (per endpoint), tier-based (per tenant plan), or learned (a small classifier picks).
  • Fail over on errors. If the primary returns a 5xx or times out, retry on a fallback (another region, another provider, or a comparable model) — but within a budget so you don’t stack retries forever.
  • Timeouts & retry budgets. Give each attempt a per-attempt timeout and cap total attempts (e.g. 2) and total wall-clock. Use exponential backoff with jitter. A retry without a budget turns a provider blip into a self-inflicted outage (a retry storm).
Retries need idempotency

If you retry a request that may have already partially succeeded, you risk doing the work (and billing) twice. Pair retries with an idempotency key so a duplicate is recognised and de-duplicated rather than re-executed. More on this under reliability below.

Queueing & backpressure for spiky load

LLM traffic is bursty and inference is slow and expensive, so demand routinely exceeds the throughput your provider quota allows. You have three honest options when that happens, and a real platform uses all three:

  • Shed — reject over-limit requests immediately with 429 + Retry-After. Best for interactive, latency-sensitive calls where a stale answer is useless.
  • Queue — admit the request to a bounded queue and process it when capacity frees up. Best for async/batch work (a nightly summarisation job) where completion matters more than instant response. Make the queue bounded so it provides backpressure instead of growing without limit.
  • Stream — for interactive requests, stream tokens as they’re generated so the user sees progress immediately, which keeps perceived latency low even when the full response is slow.

The unifying principle is admission control: decide at the front door whether to serve now, serve later, or refuse — never accept unbounded work you can’t finish. Async APIs (submit a job, poll or get a webhook for the result) are how you expose the “serve later” path cleanly to clients.

Observability

You cannot operate a platform you can’t see. Three pillars, plus the LLM-specific dimensions:

PillarAnswersFor an LLM platform, include…
LogsWhat happened on this one request?tenant_id, model, status, token counts, cache hit/miss, latency — never the raw prompt/response unless privacy-cleared.
TracesWhere did the time go across stages?spans for auth → limit → cache → provider call, so you can see whether latency is yours or the provider’s.
MetricsHow is the system behaving in aggregate?latency p50 / p99, error rate, RPM/TPM, tokens, cost, cache hit rate — sliced per tenant.

Watch p99, not just the average — averages hide the slow tail that customers actually feel. Per-tenant dashboards let you answer “is the platform slow, or is this one customer hammering a 200k-token prefix?” and back the SLOs you promise.

Gotcha: logging raw prompts and responses

Prompts and completions often contain customer PII or secrets. Logging them verbatim turns your observability store into a compliance liability and a breach blast-radius. Log metadata (token counts, latency, status, hashes) by default; gate raw content behind explicit consent, redaction, and short retention.

Reliability

The platform sits in front of a slow, occasionally-failing dependency (the model provider). Reliability is mostly about containing that failure so it doesn’t become your failure.

  • Timeouts everywhere. Every outbound call has a deadline. A request with no timeout is a thread (and a customer) hung forever.
  • Retries with budgets + backoff. Retry transient errors, but cap attempts and total time, and use exponential backoff with jitter to avoid synchronised retry storms.
  • Idempotency keys. The client sends a unique key per logical request; the gateway de-duplicates so a network retry doesn’t double-execute or double-bill. Essential for any non-read operation.
  • Circuit breakers. When a provider is clearly down, stop hammering it — trip the breaker, fail fast to the fallback, and probe periodically. This protects both you and the recovering provider.
  • Graceful degradation. When the strong model is unavailable, fall back to a weaker one and say so, rather than returning a hard error. A slightly worse answer beats no answer for most product surfaces.
The whole platform in one sentence

A stateless gateway authenticates the caller, resolves the tenant, checks request- and token-based limits against shared counters, looks up a tenant-scoped prompt cache, routes to the cheapest capable model with timeouts and a fallback, streams tokens back, meters the real token usage to a durable ledger, and emits per-tenant logs/traces/metrics — so the product is fast, fair, cheap, observable, and survives a provider going down.

How this shows up in the interview

Q1 · “Design the LLM API platform.”

Start with the spine, then layer concerns onto the gateway. “Client to a stateless gateway to model providers, with tokens streaming back. The gateway is the choke point that owns the cross-cutting work, and shared state lives beside it.” Then walk the request path in order: auth → tenant resolution → rate limit (req + tokens) → prompt-cache lookup → model routing with timeout/fallback → admission/queue → stream → meter usage. Call out the two stores: a fast in-memory store for counters and cache, a durable log/warehouse for usage. Close with cross-cutting concerns: observability (p50/p99, error rate, cost, per tenant) and reliability (timeouts, retries-with-budget, idempotency, graceful degradation). The structure itself is the answer — interviewers want to see you put rejections cheap-and-early and metering accurate-and-late.

Q2 · “How do you rate-limit a multi-tenant platform by tokens?”

Two ceilings, per tenant, in shared state. “I limit both RPM and TPM, because LLM load scales with tokens, not requests.” Use a token bucket per tenant (burst-tolerant, cheap state) stored in Redis so every stateless gateway sees the same count. Because output tokens aren’t known up front, estimate cost at admission (known input + max_tokens bound), reserve the budget, then reconcile against the provider’s real counts on completion. Over-limit → 429 with Retry-After (explicit backpressure). Per-tenant buckets give fairness; add a global ceiling to protect the upstream provider from the sum of all tenants. Mention the trap: limiting RPM alone lets one big-context tenant blow the budget while “under the limit.”

Q3 · “Design the prompt-caching layer.”

Separate the stable prefix from the dynamic tail and cache the prefix. “Most prompts are a big fixed head — system prompt, tools, shared docs — plus a small per-request tail. I cache the head.” Cache key = hash(tenant_id + model + prefix + version) — tenant- and model-scoped so nothing leaks across customers and the cached KV state matches the model. Invalidation is time-based (idle expiry) and content-addressed (any prefix change misses), with a version bump when the system prompt changes. Be explicit about the win (skip the prefix prefill → lower TTFT and discounted prefix tokens, tying back to the KV cache from the inference module) and the gotcha (volatile data at the top of the prompt kills the hit rate — keep stable content first). If pushed, note you can layer an exact-match response cache for identical full requests, but only within a tenant.

Check your understanding

Why limit by tokens as well as by requests? Isn’t requests-per-minute enough?

No — for an LLM platform, load and cost scale with tokens, not request count. A single request with a 200k-token context can cost and load the system more than a thousand tiny requests. If you only cap RPM, one tenant sending huge-context calls can exhaust your provider budget and wreck latency for everyone while technically staying under the request limit. So you enforce both RPM and TPM (often splitting input vs output tokens), estimating cost at admission and reconciling against the provider’s real counts afterward.

A teammate proposes caching prompts with the key hash(prompt), shared across all tenants. What’s wrong?

It breaks tenant isolation. Two tenants who send the same prefix would share a cache entry, which at minimum is a timing side-channel (one can detect the other’s usage) and at worst leaks one tenant’s cached response to another. The cache key must include the tenant (and the model, since cached KV state is model-specific): hash(tenant_id + model + prefix + version). Cross-tenant sharing should only happen for deliberately public, audited content — never as the default.

Your platform retries failed provider calls. A customer complains they were billed twice for one action. What did you miss?

Idempotency. A retry of a request that already partially or fully succeeded re-executed the work and re-metered it. The fix: the client supplies an idempotency key per logical request, and the gateway de-duplicates — returning the original result (and the original usage record) instead of running and billing it again. Pair this with bounded retry budgets and backoff-with-jitter so retries don’t become a storm, and a circuit breaker so you fail fast when the provider is clearly down.
